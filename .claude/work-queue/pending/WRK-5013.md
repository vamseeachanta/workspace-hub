## Mission

Fix `scripts/analysis/session-analysis.sh` so it runs without arithmetic errors on
Windows MINGW64 Git Bash by stripping embedded newlines from `wc -l` command substitutions.

## What

`wc -l` on MINGW64 Git Bash returns output with a trailing newline that survives command
substitution: `$(wc -l < file)` → `"0\n0"` instead of `"0"`. This breaks:

- `[[ $VAR -gt 0 ]]` — syntax error (token `"0"` after newline)
- `printf "%d" $VAR` — invalid number
- Arithmetic expressions: `$(( A + B + $VAR ))` — expression spans two lines

Observed at lines 151, 220, 274, 310, 319 during all 9 date analyses run 2026-03-11.

## Why

licensed-win-1 sessions generate signal files locally. Running `session-analysis.sh` on
Windows allows pre-processing before pushing to dev-primary. Currently every run produces
5+ error lines per date, degrading confidence in output and cluttering CI logs.

## Acceptance Criteria

1. `wc -l` variables stripped with `| tr -d '[:space:]'` or `xargs` at all assignment sites
2. No `syntax error` or `printf: invalid number` output on MINGW64 for any of the 9 test dates
3. TDD: `bash scripts/analysis/tests/test-session-analysis-windows.sh` — all cases PASS
4. Idempotent re-run: existing analysis files skipped cleanly (already works)
5. Script still produces correct output on Linux (dev-primary regression check)

## Root Cause Pattern

```bash
# BAD — captures trailing newline on MINGW64
N_SKILLS=$(wc -l < "$file")
# [[ $N_SKILLS -gt 0 ]]  → syntax error

# GOOD — strip whitespace
N_SKILLS=$(wc -l < "$file" | tr -d '[:space:]')
```
