## Mission

Refine `/whats-next` output to show more meaningful, assimilating results with standard enriched columns instead of prose commentary. Build on WRK-5048 (box-drawing tables).

## What

Add richer columns to whats-next tables by pulling data from WRK frontmatter and wrk-status-index.json:

### Current columns (keep)
- ICON, WRK, PRI, TITLE

### New columns to add
- **CATEGORY** — from frontmatter `category` field
- **SUBCATEGORY** — from frontmatter `subcategory` field (already parsed but not displayed)
- **URGENCY** — from urgency_score.py (already computed, stored in WRK_URGENCY associative array)
- **GH#** — GitHub issue number extracted from `github_issue_ref` frontmatter field
- **AGE** — days since `created_at` (e.g. "7d", "3mo")
- **STAGE** — current checkpoint stage (already parsed for working section, extend to ready sections)

### Additional: WRK / GitHub issue cross-index
- Surface github_issue_ref data in a queryable way
- Consider a `--with-issues` flag or always show GH# column when available

## Why

Current output shows WRK IDs and titles but lacks the metadata needed to make quick decisions (how old, how urgent, what domain, is there a GH issue). User must read individual WRK files to assess. Previous session (WRK-5048) established the table framework; this adds the data richness.

## Prior Art

- WRK-5048: established draw_table() box-drawing framework
- wrk-status-index.json: 777 entries with status, machine, title, priority, category
- urgency_score.py: already computes per-WRK urgency scores
- github_issue_ref: field exists in some WRK frontmatter (WRK-5101 wired auto-creation)
- WRK-5048 future-work.yaml: terminal width detection (FW-1), --no-color (FW-2), sorting (FW-3)

## Performance Constraint

Target: **≤2s** total execution time (current: ~26s, bottleneck is `uv run` for urgency_score.py at 21s).

### Profiling results (2026-03-20, 419 WRK files scanned)

| Component | Time | Fix |
|---|---|---|
| `uv run --no-project python urgency_score.py` | 21s | Replace with `python3` or pre-cache in index |
| grep per file × 419 | ~2s | Read from wrk-status-index.json instead |
| `find` for archive checks | 0.8s | Use index `status=archived` lookup |
| wrk-status-index.json read | 0.14s | Already fast — make this primary source |

### Strategy: index-first architecture
**Everything pre-computed at write time. whats-next is read-only.**

1. Enrich `wrk-status-index.json` to include ALL display fields:
   status, machine, title, priority, category, subcategory, urgency_score,
   github_issue_ref, created_at, blocked_by, type, note, not_before,
   checkpoint_stage, session_pid, standing, cadence
2. Index rebuild script computes urgency scores and writes them into the index
3. Every WRK mutation (add, stage transition, archive) updates the index entry
4. `whats-next.sh` reads index JSON once → filter/sort/render. Zero file scanning.
5. No `uv run`, no `grep` per file, no `find` for archive checks at query time
