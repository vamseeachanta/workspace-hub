## Mission

Eliminate the dual-numbering problem where WRK-NNN and GitHub issue #NNN are independent sequences, causing confusion for humans and agents when cross-referencing work.

## Problem

- WRK IDs are assigned by `next-id.sh` scanning local pending/ files — completely disconnected from GitHub issue numbers.
- Machine-partitioned ranges (dev-primary 1–4999, licensed-win-1 5000+) add further divergence.
- Humans and agents must mentally map between two numbering systems when referencing work in commits, PRs, and conversation.
- Risk of referencing the wrong item when numbers overlap (e.g., WRK-42 vs GitHub #42 are unrelated).

## Solution Space (evaluate in planning)

### Option A: GitHub-issue-first numbering
- Every `/work add` creates a GitHub issue first via `gh issue create`, then uses the returned issue number as the WRK ID.
- Pros: single source of truth, no mapping needed, GitHub cross-refs work natively.
- Cons: requires network access at capture time, couples local workflow to GitHub availability.

### Option B: Sync on commit/push
- Keep local WRK numbering but create a GitHub issue at stage exit (e.g., Stage 3) and store the mapping in frontmatter (`gh_issue: 123`).
- Pros: offline capture still works, eventual consistency.
- Cons: two numbers still exist during early stages.

### Option C: Reserve GitHub issue ranges
- Use GitHub API to reserve issue numbers matching the machine-partitioned WRK ranges.
- Pros: numbers stay aligned.
- Cons: fragile, wastes GitHub issue numbers.

### Option D: Abandon WRK prefix, use GH-NNN
- Replace WRK-NNN with GH-NNN everywhere, making GitHub the canonical ID source.
- Pros: simplest mental model.
- Cons: large migration effort, breaks all existing references.

## Acceptance Criteria

- [ ] Decision: which option to implement (or hybrid)
- [ ] No dual-number confusion for new items after implementation
- [ ] Existing WRK references remain valid (migration or mapping)
- [ ] Works across all machines (dev-primary, licensed-win-1, macbook-portable)
- [ ] Agents can resolve WRK↔GitHub references unambiguously
- [ ] `next-id.sh` and capture scripts updated accordingly
