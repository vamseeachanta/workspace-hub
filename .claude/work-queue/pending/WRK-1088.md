## Mission
Establish a consistent ETL pattern for all external engineering datasets so data
ingestion is reproducible, auditable, and incrementally updateable — not ad-hoc
per-module scripts.

## What
- Framework: `scripts/data/pipeline/` — a lightweight ETL harness (no heavy deps):
  - `extract.py`: base class with `fetch()` → raw bytes/JSON, `cache_path()` → local cache
  - `transform.py`: base class with `transform(raw)` → `pd.DataFrame`, schema validation
  - `load.py`: base class with `load(df)` → writes to parquet/CSV at standard path
  - `pipeline.py`: orchestrates extract→transform→load; incremental (skip if cache fresh)
- Implementations for active datasets:
  - `pipelines/eia_production.py`: EIA API → production data (worldenergydata)
  - `pipelines/bsee_wells.py`: BSEE public data → well records (worldenergydata)
  - `pipelines/yfinance_prices.py`: yfinance → price history (assethold) — with
    deterministic fixture fallback from WRK-1062
- Run: `scripts/data/pipeline/run-pipeline.sh <pipeline-name> [--force-refresh]`
- Manifest: `config/data/pipeline-manifest.yaml` — pipeline name, source URL,
  refresh cadence, last-run timestamp, record count

## Why
worldenergydata has EIAApiClient and CountryProductionLoader but each is a one-off
pattern. assethold uses yfinance directly in tests (causing the 12 live-data failures
fixed in WRK-1062). A shared ETL framework standardises: caching, schema validation,
incremental refresh, and audit logging — making all data ingestion reliable and testable.

## Acceptance Criteria
- [ ] `extract/transform/load` base classes in `scripts/data/pipeline/`
- [ ] `eia_production.py`, `bsee_wells.py`, `yfinance_prices.py` pipelines implemented
- [ ] `run-pipeline.sh <name>` executes full ETL with incremental skip if cache fresh
- [ ] `pipeline-manifest.yaml` updated on each successful run
- [ ] yfinance pipeline uses fixture fallback (WRK-1062 pattern) in test mode
- [ ] Schema validation fails loudly on unexpected source format changes
- [ ] Cross-review (Codex) passes
