# Skill upkeep and curation — evals, A/B tests, daily cron integration

## What

Add a structured eval/test harness for workspace-hub skills so that skill quality is
continuously validated. Each skill gets capability evals (does it do what it claims?) and
procedural evals (does it follow the correct workflow?). A/B variants test alternative
skill prompts against each other. All evals run nightly in the comprehensive-learning
cron pipeline.

**Skill curation principles added (2026-03-06):**

1. **Single source of truth**: each skill lives in exactly one location; no duplicates across
   `.claude/skills/` subdirectories. Cron detects and flags duplicate skill names.
2. **External skill discovery**: before building new skills, scan official upstream repos for
   reusable candidates:
   - `https://github.com/openai/skills/tree/main/skills/.curated`
   - `https://github.com/anthropics/skills`
   Cron runs a weekly diff: local skill names vs upstream catalog → surfaces import candidates.
3. **Curation cron job**: skills should be single source of truth — all of the above
   (evals, A/B tests, duplicate detection, external discovery) rolls into one
   `scripts/cron/skill-curation-nightly.sh` (or a phase of comprehensive-learning).

## Why

Skills currently have no automated quality checks. Skill regressions (prompt drift, broken
tool calls, outdated step sequences) go undetected until a session fails. A daily eval run
catches degradation early and provides objective signal for skill improvement.

## Acceptance Criteria

- [ ] Eval framework design documented (`specs/skills/skill-eval-framework.md`)
- [ ] Capability eval schema defined: each skill declares expected outputs/behaviors
- [ ] Procedural eval schema defined: step-sequence fidelity checks
- [ ] A/B eval scaffold — three comparison modes:
  1. **With vs without**: same task, skill injected vs bare model — measures whether the skill adds value
  2. **Skill vs skill**: two closely related skills run on the same task — identifies overlap, merge candidates, or which variant wins
  3. **Variant vs variant**: two prompt versions of the same skill — picks the better prompt
- [ ] At least 3 pilot skills have evals written (e.g. `work-queue`, `workflow-gatepass`, `comprehensive-learning`)
- [ ] `scripts/skills/run-skill-evals.sh` executes all evals and emits a JSONL report
- [ ] Comprehensive-learning cron (`scripts/cron/comprehensive-learning-nightly.sh`) calls the eval script
- [ ] Eval results surfaced in `/daily-report` or `/reflect` output
- [ ] Failing evals create follow-up WRK items automatically
- [ ] **Single source of truth**: cron detects duplicate skill names across subdirs and flags them
- [ ] **External discovery**: weekly scan of `openai/skills/.curated` + `anthropics/skills`; diff against local skills emits import-candidate list as WRK suggestions
- [ ] All curation tasks unified under `scripts/cron/skill-curation-nightly.sh` (or comprehensive-learning phase)
- [ ] **Retirement signal**: when A/B delta (with vs without) falls below a threshold,
  cron flags the skill as a retirement candidate and creates a WRK suggestion (never auto-deletes)
- [ ] **Auto-enhancement**: eval results feed back into `comprehensive-learning` Phase 7
  (code-quality scan) so underperforming skills get enhancement candidates queued
- [ ] Research-driven updates: cron checks for new model capabilities or published
  prompting research and proposes targeted skill enhancements as WRK items

## Agentic AI Horizon

- Skill evals are directly analogous to LLM evals — this is foundational infrastructure
  for measuring and improving agent capability over time.
- A/B skill testing lets us objectively compare prompt variants, which will become more
  valuable as model capability increases.
- Investing now builds the measurement layer needed to leverage future agentic improvements.
- Park the auto-remediation side (agent self-heals broken skills) as a future WRK once the
  measurement layer is stable.
