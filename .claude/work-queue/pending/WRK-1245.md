## Mission

Extract maximum engineering value from the entire 1M+ document corpus: LLM summaries
for 426K text-extractable documents, deep extraction (tables, charts, constants,
equations) from high-value subsets, methodology capture for calculation implementations,
and test case generation from worked examples and reference data.

## Execution Strategy

### Phase 1: Tier 1 — Standards + Specs (4,685 docs, ~$9)

**ace_standards** (3,969) + **workspace_spec** (716) — high value, low cost.
- LLM summaries via `phase-b-claude-worker.py --source ace_standards`
- Deep extraction on engineering standard PDFs (tables, constants)
- Promote extracted tables to `data/standards/promoted/`

### Phase 2a: Deep Extraction — Machine-Readable ($0 LLM)

Run `deep-extract.py` on machine-readable docs. Deterministic — tables, charts, metadata.
**WRK-1246 yield assessment** informs priority order:

| Priority | Segment | Docs | Table Yield | Est. Tables |
|----------|---------|------|-------------|-------------|
| 1 | ace_project XLS/XLSX | 11,741 | 93% | 25K |
| 2 | ace_project PDF (machine) | ~101K | 80% | 1.08M |
| 3 | dde_project PDF (machine) | ~110K | 69% | 1.67M |
| 4 | ace_standards >1MB (machine) | ~1,500 | 53% | 69K |
| 5 | dde_project DOCX/DOC | 25,902 | 26% | 38K |

- Output: tables → CSV, charts → metadata YAML
- Estimated time: 10 shards × ~2 hours = overnight batch
- Key value: S-N curves, design factors, material properties, coefficients
- Readability enrichment running (background job, `--resume` safe)

### Phase 2b: OCR Pipeline (~94K scanned PDFs)

**~94K project PDFs (27-30%) are scanned images** with no embedded text layer.
These need OCR before any text/table extraction.

- ace_project: ~47K OCR docs (30% of ace PDFs)
- dde_project: ~46K OCR docs (27% of dde PDFs)
- ace_standards >1MB: ~300 OCR docs (16%)
- Tools: tesseract (free, local), doctr (free, GPU), azure-doc-intelligence (paid, highest quality)
- Estimated compute: TBD after tool benchmarking on 50-doc sample
- This is a separate child WRK — blocked by Phase 2a completion

### Phase 3: Methodology Extraction

For each engineering domain, identify documents containing calculation procedures:
1. **Filter** by discipline + keywords ("shall be calculated", "formula", "equation")
2. **Extract** inline calculation procedures from standard text
3. **Map** to existing/needed Python functions in digitalmodel/assetutilities
4. **Create** calc-report YAMLs with inputs from extracted tables
5. **Generate** pytest test stubs from reference data

Priority domains (existing code to validate against):
- Cathodic protection (DNV-RP-B401, ISO 15589) → `digitalmodel/cathodic_protection/`
- Fatigue (DNV-RP-C203, BS 7608) → `assetutilities/calculations/`
- Pipeline stability (DNV-RP-F109) → `digitalmodel/geotechnical/`
- Structural (API RP 2A, DNV-OS-C101) → structural-analysis skill
- Mooring (API RP 2SK) → mooring-design skill

### Phase 4: Tier 2 — Project Documents (393K docs, ~$788)

**ace_project** + **dde_project** — engineering project documents.
- Batch LLM summaries via parallel shards
- Classification into project categories (design, analysis, procurement, fabrication)
- Extract project-specific data (vessel specs, environmental data, soil data)
- Budget: ~$788 at Haiku rates, spread over multiple days ($20/day = ~40 days)

### Phase 5: Test Case Generation

From all extracted data:
1. **Constants** → Python module assertions (value range checks)
2. **Tables** → CSV fixtures for parameterized tests
3. **Equations** → Function implementations with table-driven tests
4. **Worked examples** → pytest test cases with known-answer validation
5. **Charts** → Digitized data for interpolation verification
