# chore(review): Troubleshoot submit-to-{claude,codex,gemini}.sh + Comprehensive Test Suite

## Why

The submit scripts are called by agents (Claude, Codex, Gemini) during cross-review
gates. A failure in any of these scripts causes a gate failure that blocks the entire
WRK lifecycle. Current issues and any new ones discovered during WRK-1002/1003/1004
runs need to be systematically identified, fixed, and guarded by a test suite so
cross-review success rates improve for all three providers.

This item is intentionally open-ended: it collects issues as they are discovered
across WRK-1002/1003/1004 runs and treats them as a consolidated fix+test batch.

**I-06 confirmed failure (real run, 2026-03-03):**
```
# Claude review failed or timed out (exit 1)
# Root cause: provider invocation did not return renderable structured output.
# Transport mode: isolated temp directory + print/json schema
# File: .claude/work-queue/assets/WRK-624/workflow-governance-review.html
```
Root cause: `workflow-governance-review.html` (30 KB) is written raw to the temp
content file. Claude reads raw `<div>`, `<style>`, `<script>` tags and fails to
produce the required JSON schema output. No HTML-to-text conversion happens anywhere
in the pipeline.

**I-06 fix:** detect `.html` / `.htm` extension before writing the content file;
strip tags to plain text first:
```bash
if [[ "$CONTENT_FILE" =~ \.(html?|htm)$ ]]; then
  # Strip HTML tags; preserve text content for review
  CONTENT="$(sed 's/<[^>]*>//g' <<< "$CONTENT" | sed '/^[[:space:]]*$/d')"
fi
```
Or use `python3 -c "import html, re, sys; ..."` via `uv run --no-project python`
for a more robust conversion. Add `--strip-html` flag to allow explicit override.

**I-07 confirmed failures (logs/orchestrator/claude/WRK-624-*.log, 2026-03-03/04):**
```
Error: getaddrinfo EAI_AGAIN api.anthropic.com
```
All 4 WRK-624 orchestrator log files show DNS resolution failure when `claude -p`
subprocess was invoked. Fix: add pre-flight network check (`curl -s --max-time 5
https://api.anthropic.com` or `getent hosts api.anthropic.com`) before the retry
loop; emit `WARN: network unavailable` and exit gracefully (not via timeout) if
DNS fails.

**I-08 / I-09 confirmed failures (same logs):**
```
EACCES: permission denied, open '/home/vamsee/.claude/debug/...'
EACCES: permission denied, mkdir '/home/vamsee/.claude.lock'
```
`claude -p` subprocess writes debug artifacts to `~/.claude/debug/` and acquires
`~/.claude.lock`. When multiple Claude instances run concurrently the lock conflicts.
Fix I-08: `mkdir -p ~/.claude/debug` with correct permissions in script prologue or
document that the calling user must own `~/.claude/`. Fix I-09: the lock contention
is a Claude CLI design constraint — document it and add a `--max-retries 1` or
serialise cross-review invocations (run Claude → wait for exit → run Codex).

**I-10 confirmed failure + fix (live probe, 2026-03-03):**
```
# First attempt (CLAUDECODE set — default when running inside Claude Code session):
Error: Claude Code cannot be launched inside another Claude Code session.
Nested sessions share runtime resources and will crash all active sessions.
To bypass this check, unset the CLAUDECODE environment variable.
→ exit 1

# Second attempt (CLAUDECODE="" unset):
→ REQUEST_CHANGES verdict, full structured JSON output, exit 0
```
Root cause: `CLAUDECODE` env var is inherited by all child processes of a Claude Code
session. The `claude -p` subprocess detects it and refuses to launch. This is the
**primary blocker** for agent-invoked cross-reviews. I-06 (HTML input) was a symptom —
the original WRK-624 failures were actually blocked at I-10 first.

**I-10 ✓ fix:** unset `CLAUDECODE` in the script before the `claude` invocation:
```bash
# Add immediately before run_claude_once() call
export CLAUDECODE=""
```
This is safe: the subprocess needs to run as an independent `claude -p` non-interactive
session. Clearing `CLAUDECODE` does not affect the parent Claude Code session.

**I-01 ✓ fix (python3 → uv):**
```bash
# Before
python3 "$BUNDLE_BUILDER" --input "$CONTENT_FILE"
python3 "$RENDERER" --provider claude --input "$raw_file"

# After
uv run --no-project python "$BUNDLE_BUILDER" --input "$CONTENT_FILE"
uv run --no-project python "$RENDERER" --provider claude --input "$raw_file"
```

**I-02 ✓ fix (setsid fallback):**
```bash
# Replace hard-exit with warn+continue
if command -v "${SETSID_CMD:-setsid}" >/dev/null 2>&1; then
  SETSID_CMD="${SETSID_CMD:-setsid}"
else
  echo "WARN: setsid not found — running without process-group isolation" >&2
  SETSID_CMD=""
fi
# In run_claude_once: ${SETSID_CMD:+"$SETSID_CMD"} timeout ...
```

**I-07 ✓ fix (pre-flight network check):**
```bash
# Add after env validation, before retry loop
if ! getent hosts api.anthropic.com >/dev/null 2>&1; then
  echo "WARN: DNS resolution failed for api.anthropic.com — skipping claude review" >&2
  echo "# Claude review skipped — network unavailable (EAI_AGAIN)"
  exit 0
fi
```

**I-08 ✓ fix (debug dir permissions):**
```bash
# Add in script prologue (after REPO_ROOT)
mkdir -p "${HOME}/.claude/debug" 2>/dev/null || true
```

**I-11 / I-05 ✓ fix (gemini bare python3 → run_renderer helper):**
Added `run_renderer()` to `submit-to-gemini.sh` matching the codex pattern
(uv-first, python3 fallback, exit 127 if neither available).

**I-09 note (concurrent lock contention):**
`~/.claude.lock` is managed by the Claude CLI itself. The script cannot fix this
directly. Mitigation: serialise submit-script invocations in cross-review wrappers
(call Claude, wait for exit code, then call Codex/Gemini). Document in troubleshoot-log.md
that concurrent Claude subprocess runs are unsupported.
