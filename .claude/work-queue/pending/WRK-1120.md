## Mission

Make WRK ID assignment atomic so two concurrent terminals cannot claim the same ID
when creating new work items.

## What

Today's incident: two terminals both called `next-id.sh`, got `1118`, and wrote
different content to the same file — one overwrote the other silently.

WRK-1049 fixed the *claim* collision (two sessions processing the same existing item).
This fixes the *creation* collision (two sessions assigning the same new ID).

### Root Cause

`next-id.sh` only *reads* the highest existing ID and returns the next integer.
It does not *reserve* that ID. Between the read and the write, any other session can
call `next-id.sh` and get the same answer.

### Fix

Update `next-id.sh` to atomically reserve the ID it returns by immediately creating
an empty sentinel file `WRK-NNN.md` in `pending/` using bash `noclobber` (`set -C`).
If the file already exists (another session beat us), increment and retry (up to 5
attempts). Return the claimed ID and leave the sentinel file in place for the caller
to overwrite with full content.

Callers that write the full YAML frontmatter with `cat >` or `bash -c "cat > ..."` will
overwrite the sentinel — so no caller changes needed. Just the reservation logic in
`next-id.sh`.

## Acceptance Criteria

1. `next-id.sh` creates a sentinel `pending/WRK-NNN.md` before printing the ID.
2. If the sentinel already exists, `next-id.sh` auto-increments and retries (≤5 tries).
3. After 5 failed attempts, `next-id.sh` exits 1 with a clear error message.
4. Two concurrent `next-id.sh` calls always return different IDs.
5. Existing callers (`/work add`, `generate-index.py`) require no changes.
6. TDD: test script spawns two parallel `next-id.sh` subshells; asserts different IDs.
