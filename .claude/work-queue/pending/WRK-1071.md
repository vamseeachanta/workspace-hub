## Mission
Establish performance baselines for compute-heavy engineering calculations so
regressions are caught before they reach production reports.

## What
- Framework: `pytest-benchmark` added to tier-1 repos that have heavy calc modules
  (assetutilities: CP calcs; digitalmodel: structural solvers; worldenergydata: EIA aggregations)
- Benchmark suite: `tests/benchmarks/` per repo — one benchmark per major calculation
  entry point (e.g. `bench_cp_anode_design`, `bench_eia_production_aggregation`)
- Script: `scripts/testing/run-benchmarks.sh [--repo <name>]`
  - Runs pytest-benchmark, outputs JSON to `scripts/testing/benchmark-results/`
  - Compares against stored baseline; flags regressions >20% slowdown
- Baseline storage: `config/testing/benchmark-baseline.json` (committed; updated
  explicitly via `--save-baseline` flag)
- Integration: weekly cron run; WRK-1057 repo-health shows last benchmark status

## Why
Engineering calculations power client reports. A 10× slowdown in CP anode design
from a refactor may go unnoticed until a report run times out. No benchmark
infrastructure currently exists in any tier-1 repo.

## Acceptance Criteria
- [ ] `pytest-benchmark` added to assetutilities, digitalmodel, worldenergydata
- [ ] `tests/benchmarks/` with ≥1 benchmark per heavy module per repo
- [ ] `run-benchmarks.sh` runs suites, compares to baseline, flags >20% regression
- [ ] `benchmark-baseline.json` populated from initial run
- [ ] Weekly cron entry for benchmark runs
- [ ] Cross-review (Codex) passes
