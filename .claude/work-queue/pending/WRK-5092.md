## Mission

Enhance the sabithaandkrishnaestates repo structure so that property-level documents are discoverable, cross-referenced, and indexed — enabling efficient tax preparation and ongoing property management.

## What

Current anatomy review (fd_15645_westpark as reference property):

### Strengths
- Good lifecycle phases: `acquisition/` → `due_diligence/` → `ownership/` → `finance/` → `maintenance/`
- Financial actuals organized by year: `finance/actuals/2025/{taxes,insurance,maintenance,bank_deposits}`
- Due diligence well-structured: `01_financial/`, `02_legal/`, `03_physical/`

### Problems Found During Tax Prep (WRK-5086)
1. **No property-level index/README** — no single file summarizing property status, key dates, financials
2. **Insurance docs in 3+ locations**: `acquisition/closing/insurance/`, `ownership/insurance/`, `due_diligence/01_financial/insurance/`, `finance/actuals/2025/insurance/` — unclear which is current
3. **Tax docs split**: `taxes/2025/` (repo root, corporate-level) vs `finance/actuals/2025/taxes/` (property-level) — no cross-reference
4. **No depreciation schedule file** — critical for Form 8825 and ongoing tracking
5. **Lease summary scattered**: `lease_agreement_markdown.md`, `lease_reimbursement_summary.md`, `lease_term_summary.md` all in `due_diligence/02_legal/lease_documents/` — no consolidated "current lease status" view
6. **No YAML structured data for key property facts** (purchase price, closing date, tenant, lease terms, NOI, cap rate, assessed values) — all buried in markdown prose
7. **Closing docs mixed**: `acquisition/closing/` has both executed docs and pre-closing analysis (contract comparisons, estimates)

### Proposed Enhancements
1. **Property index YAML** (`investments/cre/fd_15645_westpark/property.yaml`) — single structured source for: purchase price, closing date, tenant, lease terms, HCAD values, insurance policy, mortgage status, key contacts
2. **Depreciation schedule** (`finance/depreciation-schedule.yaml`) — tracks basis, annual depreciation, accumulated depreciation per year
3. **Canonical document locations** (README.md per property):
   - Current insurance → `ownership/insurance/` (historical in `acquisition/`)
   - Current tax bills → `finance/actuals/YYYY/taxes/`
   - Corporate tax filings → `taxes/YYYY/` (repo root)
4. **Fix doc duplication** — consolidate or symlink insurance, HOA files
5. **Template** for future properties — standardize directory layout so second property follows same pattern

## Why

Tax preparation for WRK-5086 required reading 15+ files across 6 directories to gather basic property data. A property.yaml index and clear canonical locations would make this a 2-file lookup. As more properties are acquired, this pattern must scale.

## Acceptance Criteria

1. `property.yaml` exists for fd_15645_westpark with all key structured data
2. `finance/depreciation-schedule.yaml` tracks basis and annual depreciation
3. Property-level README.md documents canonical file locations
4. Template directory structure documented for future property acquisitions
5. No orphaned/duplicate documents — each doc lives in one canonical location
