# feat(session-analysis): Close the Loop — Session Analysis End-to-End

## Mission

The session analysis infrastructure (hooks → signals → cron → comprehensive-learning →
feedback) is **architecturally complete but data-dark**. Signals are being written but
the nightly pipeline reads 0 sessions every run due to a file naming mismatch. Meanwhile
three related WRK items (WRK-636, WRK-1018, WRK-1025) are individually small but
collectively describe the same "make it real" problem. This WRK fixes the pipeline,
validates every phase with real data, and adds the missing wiring to close the loop.

## Root Cause (confirmed)

`session-analysis.sh` looks for `${ANALYSIS_DATE}-*.jsonl` (timestamp-suffixed, old format)
but `emit-session-quality-signals.sh` writes to `${ANALYSIS_DATE}.jsonl` (no suffix,
current format). The early 2026-02-20 files (`2026-02-20-091148.jsonl` etc.) match the
old pattern; daily consolidated files (`2026-03-09.jsonl`) do not. Result: 297 real
signals/day, 0 detected.

## What

### Layer 1: Fix the signal naming mismatch (Critical — 1-line fix)

In `scripts/analysis/session-analysis.sh` Step 2, update the `find` pattern to match
both formats:
```bash
# Old: name "${ANALYSIS_DATE}-*.jsonl"
# New: name "${ANALYSIS_DATE}*.jsonl"
```
Verify the fix by running `session-analysis.sh` manually and confirming N_SESSIONS > 0.

### Layer 2: Validate comprehensive-learning Phase 1-10 with real data

Run `comprehensive-learning-nightly.sh` manually (`bash --dry-run` or staged) and verify:
- Phase 1 (Insights): extracts from the now-visible signal files
- Phase 4 (Improve): produces actual memory/rule/skill diffs
- Phase 7 (Action Candidates): converts `candidates/script-candidates.md` entries to WRK items
- Phase 10 (Report): writes coherent learning report to `session-reports/`

Document any phase that is still a stub (no-op) and capture follow-on WRKs.

### Layer 3: Add skill invocation signal (WRK-305 follow-on)

Add `event: skill_invoked` signal to `emit-session-quality-signals.sh` or a dedicated
hook. The signal should include: `skill`, `session_id`, `wrk`, `ts`.

Parse basis: session transcript contains `[Skill tool]` invocations — grep for skill name
from `stop_hook_active` transcript context.

### Layer 4: Wire Codex session ingestion (absorbs WRK-636)

Extend `comprehensive-learning-nightly.sh` to ingest Codex session JSONL files:
- Source: `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl`
- Format: `{"timestamp": "...", "type": "response_item", "payload": {...}}`
- Action: extract WRK references, error patterns, skill hints from Codex sessions
- Append to the same Phase 1 input so Codex learnings feed the same pipeline

Windows (acma-ansys05): Codex sessions live at equivalent Windows path — include only
if rsync from acma-ansys05 is working; otherwise defer to follow-on.

### Layer 5: Surface unqueued future-work.yaml items (absorbs WRK-1018)

Extend Phase 6 (`feat(comprehensive-learning): Phase 6`) to scan recently-archived WRK
asset directories for `future-work.yaml` entries with `captured: false` and route them
into Phase 7 ActionCandidates so they appear in the next nightly queue.

### Layer 6: Execution-stage analytics (absorbs WRK-1025)

Extend `emit-session-quality-signals.sh` to emit per-stage signals at Stage 10 exit:
- Signal type: `stage_exit`
- Fields: `wrk`, `stage`, `duration_s`, `edits`, `reads`, `tool_calls`, `outcome`
- Used by Phase 1 to compute per-stage cost and detect anomalies (e.g. Stage 3 taking
  10× longer than average)

### Layer 7: Closed-loop validation test

After all layers are wired, run one full nightly cycle manually and confirm:
1. `N_SESSIONS > 0` in session-analysis.sh output
2. skill-scores.yaml has non-null `usage_rate` for at least 3 skills
3. At least 1 `skill_invoked` signal in today's signal file
4. Phase 7 converts ≥1 candidate entry to a new WRK pending item
5. `learning-reports/YYYY-MM-DD.md` exists and has non-trivial content
