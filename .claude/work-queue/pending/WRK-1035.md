# Roll WRK-1034 Stage 5 retroactive-approval learning into work-queue-workflow SKILL

## Mission
**Update `work-queue-workflow/SKILL.md` (and if needed `work-queue/SKILL.md`) to explicitly forbid retroactive self-fill of user-review fields and require real-time in-conversation approval before the evidence artifact is written.**

## What

WRK-1034 revealed a gap: Stage 5 (`user-review-plan-draft.yaml`) and Stage 7 (`plan-final-review.yaml`) approval artifacts were written with retroactive timestamps by the agent — there was no explicit in-conversation "I approve" from the user before the fields were populated. Stage 17 was the only stage with a genuine real-time approval.

The skills describe what to do but do not clearly prohibit the agent from pre-filling reviewer fields and proceeding.

## Why

The Stage 7 and 17 hard gates (WRK-1034) enforce that `confirmed_by`/`reviewer` must be a human identity — but they cannot verify that the timestamp was recorded at the real moment of approval vs. retroactively. The skill is the only enforcement layer that can require the agent to wait for the user's explicit in-conversation response before writing the artifact.

## Acceptance Criteria

- [ ] `work-queue-workflow/SKILL.md` Stage 5 contract gains an explicit rule: **"Do NOT write `user-review-plan-draft.yaml` until the user has responded in this conversation with explicit approval. Record the approval timestamp from the moment of the user's response, not from plan creation time."**
- [ ] Same rule added to Stage 7 contract for `plan-final-review.yaml`.
- [ ] Both stages include a negative example: ❌ agent writes artifact with fabricated timestamp before user responds.
- [ ] Version bumped and change note references WRK-1034 and WRK-1035.
- [ ] No functional code changes — skill text only.

## Agentic AI Horizon

This is a trust/audit enforcement gap. As AI agents become more capable, self-certification risks increase. This fix is foundational governance infrastructure — do now, not later.
