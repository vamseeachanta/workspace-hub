# WRK-1028 — Stage-isolated WRK lifecycle

## Mission
**Make every WRK stage independently executable from its predecessor's exit artifacts, with its own skill(s), entry protocol, and exit protocol — so context compaction, context rot, and stage-jumping cannot degrade workflow quality.**

> Read this at the start of every execution stage. If a task, file change, or decision is not directly in service of this mission, stop and capture it as a new WRK item instead.

## What

The current 20-stage lifecycle runs as a single long conversation. This causes:
1. **Context compaction** — hard-stop instructions compressed out; agent forgets Stage 5 gate
2. **Context rot** — 10 stages of accumulated planning context pollutes Stage 10 execution
3. **Stage jumping** — no mechanical barrier between stages; tool-call momentum skips gates
4. **Quality drift** — heavy stages (10, 12, 17) use the same depth as light stages (3, 9)

The fix is a **stage-isolation architecture**:
- Each stage is self-contained: entry reads previous stage's exit artifacts, executes, exits
- Each stage has its own `skill(s)` block — loaded fresh at entry, not carried from previous stage
- Each stage has a `weight` (light/medium/heavy) that determines required sub-skills and depth
- `start-stage.sh WRK-NNN <N>` bootstraps a fresh context: loads only stage contract + exit artifacts from Stage N-1
- `exit-stage.sh WRK-NNN <N>` validates exit artifacts, writes log, commits, signals readiness for Stage N+1
- Hookify backstops block premature stage advancement (all 3 human gates: 5, 7, 17)

## Why

Context compaction and context rot are structural, not behavioral. No amount of SKILL.md text fixes them — only isolation does. Each stage starting fresh from the previous stage's artifacts is the same principle used by CI/CD pipelines: each job gets a clean workspace built from the previous job's published artifacts, not from a shared mutable state.

This also enables:
- **Resumability**: any stage can be resumed independently after a session drop
- **Parallel execution**: Route C stages 6/8/9 can run concurrently if isolated
- **Engineering quality**: heavy stages get their own full sub-skill chain, not a subset of a compressed context

## Stage Classification

| Weight | Stages | Required skills at entry |
|--------|--------|--------------------------|
| Light | 1, 3, 8, 9, 18, 19, 20 | work-queue only |
| Medium | 2, 4, 6, 11, 13, 14, 15, 16 | work-queue + domain skill |
| Heavy | 5, 7, 10, 12, 17 | work-queue + brainstorming / TDD / systematic-debugging / workflow-html |

Human-in-loop stages: **1, 5, 7, 17** — require explicit human response before `exit-stage.sh` accepts exit artifacts.

## Stage Isolation Contract (per stage)

Each stage defined in `scripts/work-queue/stages/stage-NN-<slug>.yaml`:

```yaml
order: 5
name: User Review - Plan Draft
weight: heavy
human_gate: true
skills_required:
  - workspace-hub/work-queue-workflow    # always first
  - superpowers/brainstorming            # heavy stages only
  - workspace-hub/workflow-html          # HTML artifact stages
entry_reads:                             # files loaded from previous stage
  - evidence/resource-intelligence.yaml
  - plan-draft-review.html
exit_artifacts:                          # files that MUST exist before exit
  - evidence/user-review-plan-draft.yaml  # decision: approved required
  - plan-draft-review.html
blocking_condition: "evidence/user-review-plan-draft.yaml missing or decision != approved"
log_action: plan_draft_complete
context_budget_kb: 8                     # max context this stage may load
```

## Agentic AI Horizon

- Stage-isolated execution = each stage is a stateless function: `f(stage_contract, prev_exit_artifacts) → exit_artifacts`
- This is the exact model needed for fully autonomous WRK pipelines: orchestrator calls `start-stage.sh`, agent runs, calls `exit-stage.sh`, loop
- Context budget per stage prevents runaway context growth; agents get exactly what they need
- Route C parallel execution becomes safe: isolated stages can run in parallel Task agents without shared mutable state
- Micro-skills per stage replace prose instructions — machine-loadable, version-controlled, testable

## Design Notes (from brainstorming 2026-03-07)

**Root cause analysis — 5 failure categories:**
1. Context compaction: Stage 5 hard-stop text lost; single-session monolith grows to 200KB+
2. Context rot: Stage 10 carries forward all planning context; execution sees irrelevant noise
3. Stage jumping: tool-call momentum; no mechanical block between stages
4. Format miss: artifact contracts buried in verify-gate-evidence.py, not in agent context
5. Quality drift: all stages treated uniformly; heavy stages (TDD, review) get same depth as light stages

**Key architectural insight**: Each stage = fresh context window seeded from previous stage's exit artifacts. Same model as CI/CD job isolation. Eliminates compaction + rot at the source.

**Stage weights drive quality depth**:
- Light stages: minimal context, single skill, fast exit
- Heavy stages: sub-skill chain mandatory (brainstorming → TDD → systematic-debugging), enforced by `start-stage.sh` which checks skill invocation before proceeding

**Staged delivery (P1 → P3):**
- P1: Hookify backstops for all 3 human gates (immediate, highest-impact)
- P2: `start-stage.sh` / `exit-stage.sh` + `stage-state.yaml`
- P3: 20 per-stage YAML contracts + 20 per-stage micro-skills
- P4: Heavy-stage enforcement (TDD invocation check, context budget enforcement)
