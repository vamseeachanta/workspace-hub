# Per-session log files in session-logger.sh

## What

Modify `.claude/hooks/session-logger.sh` so each Claude session gets its own
log file instead of merging all activity for the day into one file.

## Why

When multiple sessions run on the same day (or across a long session), the
current `session_YYYYMMDD.jsonl` file merges all tool calls into one file.
Per-session files make it easier to correlate log entries with a specific
conversation, debug session issues, and feed individual sessions into the RAG
pipeline cleanly.

## Approach

Use a state file `.claude/state/current-session-id` (gitignored) to persist
the session ID across hook calls:

1. On each `PreToolUse` call, check if the state file exists and if
   `last_active` epoch is within 7200 seconds (2 hours).
2. If stale or missing → generate a new session ID:
   `session_YYYYMMDD_HHMMSS` (timestamp of first tool call).
3. Update `last_active` in the state file on every hook call.
4. Write logs to:
   - `.claude/state/sessions/session_YYYYMMDD_HHMMSS.jsonl`
   - `logs/orchestrator/claude/session_YYYYMMDD_HHMMSS.jsonl`

State file format (plain text, two lines):
```
session_id=session_20260303_143022
last_active=1740999022
```

## Acceptance Criteria

- [ ] New session → new log file with `session_YYYYMMDD_HHMMSS` name
- [ ] Resumed session (within 2h) → same log file continues
- [ ] Session idle >2h → next tool call starts a new log file
- [ ] Both log paths (state/sessions/ and logs/orchestrator/claude/) use the session ID
- [ ] State file is gitignored (already under `.claude/state/` which is gitignored)
- [ ] Hook remains <1s execution time (raw-write only principle)
- [ ] Existing date-based fans in analyze-sessions.sh or similar still work
  (glob `session_*.jsonl` already matches new names)
