## Mission

Prevent context bloat and context rot by resetting the Claude context window
at each stage boundary ‚Äî using checkpoint.yaml + subagent-per-stage execution.

## What

### Assessment

**Current state**: `/work run WRK-NNN` runs stages sequentially in a single
conversation. Context accumulates across stages 1‚Üí2‚Üí3‚Üí4‚Ä¶ consuming tokens and
risking rot (agent forgets earlier stage decisions).

**Mechanism already exists**:
- `checkpoint.yaml` records `stage`, `next_action`, `context_summary`, `entry_reads`
- `start_stage.py` reads checkpoint and prints the resume block
- `/wrk-resume WRK-NNN` reads checkpoint and loads entry_reads into context
- `exit_stage.py` validates stage exit and (partially) updates checkpoint

**Gap**: Agents continue inline rather than signalling "reset context now."

**Best approach ‚Äî Subagent-per-stage**:
Each stage is executed by a fresh Task agent (spawned by the orchestrator) that
receives only: checkpoint.yaml + entry_reads files. No accumulated conversation
history. Matches CLAUDE.md "orchestrate, don't execute" hard gate.

**Session log role**: `logs/orchestrator/claude/session_YYYYMMDD.jsonl` captures
all tool calls. The orchestrator can read the log to verify stage completion
signals (`stage_complete` event) before spawning the next stage agent.

### Implementation

**Phase 1 ‚Äî exit_stage.py writes rich checkpoint**:
- After each stage exit, write full `context_summary` (what was done, key findings, P1 items)
- Write `entry_reads` list (files the next stage agent must load)
- Include `stage_complete_at` timestamp

**Phase 2 ‚Äî work.sh stage-loop uses Task agents**:
- After each stage exit signal, orchestrator spawns a new Task agent
- Task agent reads checkpoint + entry_reads, executes next stage, exits
- Orchestrator polls for `stage_complete` log event before spawning next agent
- Human-gate stages (5, 7, 17) pause the loop and surface the blocking prompt

**Phase 3 ‚Äî stage micro-skills get entry_reads contracts**:
- Each `stage-NN-*.md` micro-skill lists exactly which files to read at entry
- These become the `entry_reads` list in checkpoint.yaml

## Why

- Context bloat slows responses and inflates token costs per session
- Context rot causes agents to forget stage decisions from 10+ messages ago
- Subagent isolation already the CLAUDE.md-mandated pattern ‚Äî stages should use it
- Session logs already capture stage completion signals ‚Äî use them

## Acceptance Criteria

- [ ] `exit_stage.py` writes `context_summary`, `entry_reads`, `stage_complete_at` to checkpoint
- [ ] `work.sh` spawns a Task subagent per stage (not inline execution)
- [ ] Human-gate stages (5, 7, 17) surface blocking prompt to orchestrator before next spawn
- [ ] Session log `stage_complete` event emitted at each stage exit
- [ ] `/work run WRK-NNN` in a new message correctly resumes from checkpoint with fresh context
- [ ] No stage evidence lost ‚Äî entry_reads covers all prior evidence files needed

## Risks

- If checkpoint is incomplete, subagent starts with wrong context ‚Üí add validation
- Hard-gate stages need orchestrator to wait for user response before spawning next agent
- Session log polling adds latency ‚Äî acceptable if < 5s per stage transition
