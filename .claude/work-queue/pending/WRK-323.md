# WRK-323 — Audit & document all OrcaFlex model parameters from modular examples

**Status:** complete | **Priority:** high | **Percent complete:** 100

## Goal

Extract every unique parameter key from `inputs/parameters.yml` across all modular examples
(A01–Z09, 49 directories), categorise them, and produce `PARAMETER_REFERENCE.md` in the
examples folder.

Analog to WRK-317 (OrcaWave parameter reference for L01–L06).

## Scope

Examples root: `digitalmodel/docs/domains/orcaflex/examples/modular/`

Categories to cover (by representative letter group):
- **A** — Riser geometry (catenary, lazy wave, pliant wave, steep wave)
- **B** — Drilling riser
- **C** — Multi-static / code check variants
- **E** — Lay table and installation automation
- **K** — Offshore wind (FOWT / spar)
- **L** — Vessel / semi-sub motions
- **Z** — Edge cases and special configurations
- Remaining groups (D, F, G, H, I, J, M) enumerated but may share parameter keys

## Output file

`digitalmodel/docs/domains/orcaflex/examples/PARAMETER_REFERENCE.md`

## Acceptance criteria

- All unique parameter keys from every `inputs/parameters.yml` listed with:
  - Type (float, int, string, list)
  - Engineering units
  - Brief description
- Parameters grouped by category: environment, line geometry, vessel/structure, load case,
  analysis settings, staging
- Parameters that differ across example categories highlighted (e.g. `water_depth`,
  `stage_durations`, `hs`, `tp`, `wave_direction`)
- Cross-reference table: which examples use which parameter keys
- Reference committed to `digitalmodel` repo

## Notes

- Use the same structure as `orcawave/examples/PARAMETER_REFERENCE.md` as a template
- OrcaFlex parameter keys follow the OrcFxAPI naming conventions (snake_case in YAML,
  PascalCase in the API object)
- Where a parameter maps directly to an OrcFxAPI attribute, note the API path
  (e.g. `water_depth` → `model.general.WaterDepth`)
