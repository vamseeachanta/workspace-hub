## Mission

Fix `next-id.sh` so it issues IDs from the current machine's range regardless of WRK files from other machines present in `pending/` or `working/`.

## What

`next-id.sh` currently scans ALL files in `pending/` and `working/` to find MAX_ID, then increments. When acma-ansys05 WRK-5000+ files exist in those directories, ace-linux-1 sessions jump to 5001+ — outside their assigned range (1-4999).

The fix: filter scanned files to only those within the current machine's floor/ceiling range (from `config/work-queue/machine-ranges.yaml`) before computing MAX_ID.

**Scope note:** Linux machines (ace-linux-1, ace-linux-2, gali-linux-compute-1) do not clash with each other — only the Windows machine (acma-ansys05, range 5000-9999) causes contamination because its WRK files appear in shared `pending/` directories.

## Why

IDs should be captured on the machine where the work is logged, irrespective of which machine the WRK is later assigned to. Cross-machine files in shared directories corrupt the ID sequence, causing ace-linux-1 to issue IDs in acma-ansys05's range.
