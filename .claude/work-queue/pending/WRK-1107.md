# assess(sessions): Unified Provider Assessment + Compliance Audit

## Mission

Consolidates WRK-1005 (provider capability assessment) and WRK-1045 (gate compliance audit)
into a single analysis pass. Read WRK-1002/1003/1004 artifacts + live session logs from
2026-03-08→2026-03-15. Emit outputs as session signals so the nightly pipeline consumes
them automatically — no ad-hoc re-analysis needed.

## Why consolidate WRK-1005 + WRK-1045

Both items read the same log sources (`logs/orchestrator/claude/`, `~/.codex/sessions/`,
`~/.gemini/tmp/`), produce provider comparison matrices, and feed the same consumers
(harness routing rules, `session-analysis.sh` Phase 1). Running them separately wastes
one full analysis pass. Emitting structured signals means future nightly runs
automatically inherit this data.

## What

### Output 1 — Provider capability report
`assets/WRK-1107/provider-assessment.md`

Dimensions (from WRK-1005 plan):
1. Lifecycle completeness — can each provider run all 20 stages without manual intervention?
2. Gate compliance — `verify-gate-evidence.py` pass rate on first attempt
3. TDD faithfulness — test-before-implementation adherence
4. User-review integration — Stage 5/7/17 gate halt correctness
5. Tool/skill invocation capability (WRK-1000 Codex limitation applies)
6. Session log completeness

### Output 2 — Gate compliance audit
`assets/WRK-1107/session-compliance-audit.md`

Analyse all WRK sessions 2026-03-08 → 2026-03-15:
- Stage 5/7/17 hard gate adherence (target: 0 violations)
- L1 hook trigger count (WRK-1044)
- Compliance score per provider vs simulation baseline (Claude=100, Codex=100, Gemini=95)
- Any regressions since WRK-1035/1044 hardening

### Output 3 — Routing rule update
`config/agents/routing-rules.yaml` (create if absent)

Update or create provider routing rules based on assessment findings.

### Output 4 — Session signal emission
Emit structured JSONL to `state/session-signals/YYYY-MM-DD.jsonl`:

```jsonl
{"event":"gate_compliance_score","provider":"claude","score":0.97,"date":"2026-03-15"}
{"event":"gate_compliance_score","provider":"codex","score":0.95,"date":"2026-03-15"}
{"event":"provider_capability","provider":"gemini","lifecycle_complete":true,"date":"2026-03-15"}
```

`session-analysis.sh` picks these up automatically in Phase 1 — no pipeline changes needed
(WRK-1102 already added the `stage_exit` and `codex_session` event types as precedent).
