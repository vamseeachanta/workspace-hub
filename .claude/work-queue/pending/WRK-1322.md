# WRK-1322: Evaluate All Workflows — Standardize on Stage-Contract Pattern

## Problem Statement

The WRK lifecycle is not the only workflow in workspace-hub. Many other workflows exist (calculation, Excel extraction, research, cross-review, code porting, etc.) but they are ad-hoc — defined as prose-only skills without declarative stages, transition hooks, checklists, or HTML visibility.

After WRK-1316 (enforcement layer) and WRK-1321 (folder-skill architecture), we have the infrastructure to standardize ALL workflows on the same stage-contract pattern: each workflow has stages, each stage has a contract (entry/exit artifacts, checklist, hooks, tools), and an orchestrator manages transitions.

## Scope

### Phase 1: Audit & Catalog
Inventory ALL workflows across workspace-hub. For each, document:
- Name and purpose
- Current stages (implicit or explicit)
- Current enforcement level (L0 prose → L3 hook)
- Where the workflow lives (skill, script, ad-hoc)
- Pain points and violations observed
- Which stages need human gates vs auto-proceed

### Known Workflows to Evaluate
- **Calculation workflow** — engineering calculations pipeline (input → validate → compute → report)
- **Excel calculation workflow** — dark intelligence extraction → validated calc → test data
- **Research-to-WRK workflow** — paper/article → distill findings → capture as WRK items
- **Cross-review workflow** — package → dispatch to 3 providers → synthesize verdicts
- **Code porting / legal scan workflow** — import code → legal scan → strip client refs → validate
- **Session start/end workflow** — init provider → load context → work → checkpoint → close
- **Document-index pipeline** — Phase A→B→C→D→E→F→G (already has stages, but no contracts)
- **Benchmark workflow** — select benchmarks → run → compare to baseline → report
- **Knowledge capture workflow** — extract learnings → validate → index → query

### Phase 2: Define Stage Contracts Per Workflow
For each workflow, define:
- Ordered stages with names
- Entry/exit artifacts per stage
- Mandatory checklists per stage
- Human gates (if any)
- Tools/skills activated per stage
- Hooks (pre_enter, pre_exit)

### Phase 3: Create Child WRKs
Each workflow gets its own WRK to implement the stage-contract pattern:
- Folder-skill per workflow (Tier 1 orchestrator + Tier 2 stage skills)
- Reuse run_hooks.py and verify_checklist.py from WRK-1316
- Reuse folder-skill pattern from WRK-1321

## Acceptance Criteria

- [ ] AC1: Workflow audit YAML listing all workflows with current enforcement levels

- [ ] AC3: Child WRKs created for each workflow implementation

## Key Insight

Every repeatable multi-step process is a workflow. Every workflow benefits from:
- Declarative stages with contracts
- Mandatory checklists
- Transition hooks
- HTML visibility
- Human gates where decisions matter

The WRK lifecycle was the first to get this treatment. Now every workflow gets the same discipline.
