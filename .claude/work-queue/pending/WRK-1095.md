# Complexity ratchet — radon cyclomatic complexity progressive gate

## Mission

Cyclomatic complexity can only decrease or hold steady — new code must not increase the aggregate complexity score per repo.

## What

Modelled on the coverage ratchet (WRK-1067) and mypy ratchet (WRK-1092):

1. **Baseline capture** — run `radon cc` per repo, record aggregate complexity stats in `config/quality/complexity-baseline.yaml` (average CC, count of functions >10 CC)
2. **Ratchet check** — `scripts/quality/check_complexity_ratchet.py`: reads baseline, runs radon, fails if high-complexity function count increased
3. **Hard gate threshold** — any single function with CC > 20 is an immediate block (not just aggregate)
4. **Pre-push gate integration** — add to existing pre-push hook
5. **`check-all.sh` integration** — upgrade `--radon` from warn-only to gated when ratchet enabled
6. **Auto-baseline update** — baseline moves forward when complexity improves

## Why

WRK-1081 added radon but as warn-only. Complexity has been growing in some repos (digitalmodel: benchmark_plotter.py 2700L, report_generator.py 2034L are flagged candidates). Without a ratchet it will continue to grow. This converts the warning into a self-enforcing gate.

## Acceptance Criteria

- [ ] `config/quality/complexity-baseline.yaml` — per-repo: avg CC, high-CC function count
- [ ] `scripts/quality/check_complexity_ratchet.py` — ratchet + CC>20 hard block
- [ ] Pre-push hook extended
- [ ] `check-all.sh --complexity-ratchet` flag
- [ ] ≥5 TDD tests (baseline capture, ratchet pass/fail, hard-block at CC>20, auto-update, report)
- [ ] Passes `check-all.sh` for the script itself
