## Mission
Implement automated secrets scanning so hardcoded API keys, tokens, and passwords
are caught before they reach git history — fulfilling the mandate in security.md.

## What
- Script: `scripts/security/secrets-scan.sh` — runs `gitleaks detect` (or
  `trufflehog filesystem`) across all tier-1 repos and workspace-hub
- Config: `.gitleaks.toml` at hub root with allowlist for known false positives
  (e.g. test fixture tokens, example placeholders)
- Pre-commit hook: add secrets-scan as a pre-commit hook step in each repo's
  `.pre-commit-config.yaml` (runs on staged files only for speed)
- Pre-push hook (WRK-1064 integration): add full-history scan for any new commits
- Output: violations with file:line, secret type, severity; exit 1 on any find
- Baseline: run once on full history; document any existing findings in
  `scripts/security/secrets-baseline.json` (gitleaks baseline format)
- CI signal: integrate scan result into repo-health dashboard (WRK-1057)

## Why
security.md mandates no hardcoded secrets and pre-commit scanning, but no tooling
implements this. A leaked API key in git history is irreversible. Baseline + hooks
closes the gap before engineering WRKs add new credentials.

## Acceptance Criteria
- [ ] `secrets-scan.sh` runs gitleaks or trufflehog across all repos
- [ ] `.gitleaks.toml` with allowlist for known false positives
- [ ] Pre-commit hook added to all repos with `.pre-commit-config.yaml`
- [ ] `secrets-baseline.json` documents any pre-existing findings
- [ ] Pre-push integration with WRK-1064 hook
- [ ] Repo-health dashboard shows secrets-clean status per repo
- [ ] Cross-review (Codex) passes
