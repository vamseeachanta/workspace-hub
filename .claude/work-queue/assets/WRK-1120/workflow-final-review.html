Prompt Start Context — Today's incident: two terminals both called `next-id.sh`, got `1118`, and wrote different content to the same file — one overwrote the other...

Today's incident: two terminals both called next-id.sh, got 1118, and wrote different content to the same file — one overwrote the other silently.

WRK-1049 fixed the claim collision (two sessions processing the same existing item). This fixes the creation collision (two sessions assigning the same new ID).

Root Cause

next-id.sh only reads the highest existing ID and returns the next integer. It does not reserve that ID. Between the read and the write, any other session can call next-id.sh and get the same answer.

Fix

Update next-id.sh to atomically reserve the ID it returns by immediately creating an empty sentinel file WRK-NNN.md in pending/ using bash noclobber (set -C). If the file already exists (another session beat us), increment and retry (up to 5 attempts). Return the claimed ID and leave the sentinel file in place for the caller to overwrite with full content.

Callers that write the full YAML frontmatter with cat > or bash -c "cat > ..." will overwrite the sentinel — so no caller changes needed. Just the reservation logic in next-id.sh.
