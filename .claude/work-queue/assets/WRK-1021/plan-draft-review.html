What "Claude Code schedule tasks" means here

The existing claude --dangerously-skip-permissions --skill <skill-name> invocations in cron ARE Claude Code scheduled tasks. The deliverable is a canonical config/scheduled-tasks/schedule-tasks.yaml that declares every task (schedule, machine, command, purpose) and becomes the single source of truth that setup-cron.sh reads, replacing its hardcoded ENTRIES arrays.

Phase 1 — Create schedule-tasks.yaml source of truth

  • File: config/scheduled-tasks/schedule-tasks.yaml (new directory)
  • Schema: one entry per task — id, label, schedule, machines[], command, log, is_claude_task, description
  • Include all 6 known jobs from ace-linux-1; mark ace-linux-2 scope; flag Windows stubs
  • Run: manual file creation

Phase 2 — Fix setup-cron.sh to read from YAML

  • File: scripts/cron/setup-cron.sh (edit)
  • Replace hardcoded ENTRIES=() arrays with uv run --no-project python reader of schedule-tasks.yaml filtered by machines + current hostname
  • Keep --dry-run flag
  • Test: bash scripts/cron/setup-cron.sh --dry-run on ace-linux-1

Phase 3 — Fix script path drift in actual crontabs

  • ace-linux-1: update 3 wrong-path entries in live crontab to match schedule-tasks.yaml canonical paths
  • ace-linux-2: add missing repo-sync; remove jobs that should only be on ace-linux-1 (or promote ace-linux-2 to full role and document rationale)
  • Create wrapper scripts/cron/skills-curation.sh (tiny exec claude ... wrapper)

Phase 4 — Document Windows tasks + tests

  • Append Windows Task Scheduler YAML stubs to schedule-tasks.yaml
  • Write scripts/cron/validate-schedule.py — parses schedule-tasks.yaml and validates cron expressions + required fields (unit test)
  • Tests: 1 test per task entry; 1 dry-run smoke test on ace-linux-1
