Assessment

Current state: /work run WRK-NNN runs stages sequentially in a single conversation. Context accumulates across stages 1→2→3→4… consuming tokens and risking rot (agent forgets earlier stage decisions).

Mechanism already exists: - checkpoint.yaml records stage, next_action, context_summary, entry_reads - start_stage.py reads checkpoint and prints the resume block - /wrk-resume WRK-NNN reads checkpoint and loads entry_reads into context - exit_stage.py validates stage exit and (partially) updates checkpoint

Gap: Agents continue inline rather than signalling "reset context now."

Best approach — Subagent-per-stage: Each stage is executed by a fresh Task agent (spawned by the orchestrator) that receives only: checkpoint.yaml + entry_reads files. No accumulated conversation history. Matches CLAUDE.md "orchestrate, don't execute" hard gate.

Session log role: logs/orchestrator/claude/session_YYYYMMDD.jsonl captures all tool calls. The orchestrator can read the log to verify stage completion signals (stage_complete event) before spawning the next stage agent.

Implementation

Phase 1 — exit_stage.py writes rich checkpoint: - After each stage exit, write full context_summary (what was done, key findings, P1 items) - Write entry_reads list (files the next stage agent must load) - Include stage_complete_at timestamp

Phase 2 — work.sh stage-loop uses Task agents: - After each stage exit signal, orchestrator spawns a new Task agent - Task agent reads checkpoint + entry_reads, executes next stage, exits - Orchestrator polls for stage_complete log event before spawning next agent - Human-gate stages (5, 7, 17) pause the loop and surface the blocking prompt

Phase 3 — stage micro-skills get entry_reads contracts: - Each stage-NN-*.md micro-skill lists exactly which files to read at entry - These become the entry_reads list in checkpoint.yaml

Why — - Context bloat slows responses and inflates token costs per session - Context rot causes agents to forget stage decisions from 10+ messages...
  • Context bloat slows responses and inflates token costs per session
  • Context rot causes agents to forget stage decisions from 10+ messages ago
  • Subagent isolation already the CLAUDE.md-mandated pattern — stages should use it
  • Session logs already capture stage completion signals — use them

Scope decision (from resource intelligence + cross-review P1 fixes)

exit_stage.py validates checkpoint but never writes it. work.sh cannot call the Task tool — only the Claude orchestrator conversation can. The full auto-loop is therefore an orchestrator-level behaviour driven by SKILL.md instructions, triggered by a machine- readable signal emitted by exit_stage.py. No bash-level Task spawning needed.

Cross-review P1 resolutions: - human_gate is read from the next stage's contract YAML (not hardcoded) - Checkpoint schema aligns with checkpoint.sh canonical: current_stage for the NEXT stage to execute (so start_stage.py resume block fires correctly), checkpointed_at for timestamp — exit_stage.py and checkpoint.sh both produce the same schema - Stage 4 uses invocation: chained_agent covering stages 2/3/4; the auto-loop honours chained_stages groups — spawns one subagent per group, not per individual stage - Stage 20 terminal: exit_stage.py stage 20 writes current_stage: complete, no STAGE_GATE - log-gate-event.sh invocation: wrk_id N stage_complete claude "Stage N complete"

Phase 1 — exit_stage.py writes rich checkpoint

After each successful validate_exit(wrk_id, N):

  1. Find next stage contract (stage-{N+1:02d}-*.yaml); if N=20, write terminal checkpoint
  2. Read human_gate from next stage contract (not hardcoded)
  3. Read entry_reads from next stage contract (already present in contracts)
  4. Read chained_stages from next stage contract if invocation: chained_agent
  5. Accept --context-summary "..." arg; when absent, auto-generate from exit artifact paths
  6. Write/overwrite checkpoint.yaml (aligned with checkpoint.sh schema): ```yaml wrk_id: WRK-NNN current_stage: N+1 # next stage to execute (matches start_stage.py expectation) stage_name: completed_stage: N # completed stage (informational) next_action: "Execute stage N+1: " context_summary: "" entry_reads:
    • .claude/work-queue/assets/WRK-NNN/checkpoint.yaml
    • checkpointed_at: # canonical field name human_gate: chained_stages: [...] # present only if invocation=chained_agent `` Terminal (N=20):current_stage: complete`, no STAGE_GATE printed.
  7. Emit log event: log-gate-event.sh WRK-NNN N stage_complete claude "Stage N complete"

Phase 2 — STAGE_GATE signal + orchestrator auto-loop

exit_stage.py prints after checkpoint write (skipped when N=20):

╔══════════════════════════════════════════════════╗
║  STAGE_GATE                                      ║
║  wrk_id:        WRK-NNN                          ║
║  completed:     N  (<stage name>)                ║
║  next_stage:    N+1  (<next name>)               ║
║  human_gate:    false                            ║
║  action:        spawn_subagent                   ║
╚══════════════════════════════════════════════════╝

human_gate: true → action: await_user_approval (no subagent spawned).

work-queue-workflow/SKILL.md gets a mandatory Stage Auto-Loop section:

On every exit_stage.py success, read STAGE_GATE: - spawn_subagent → call Agent tool immediately (see template below) - await_user_approval → print gate prompt, STOP; resume loop after user approves - No STAGE_GATE (stage 20 terminal) → WRK complete; archive

Stage subagent prompt template (in SKILL.md):

You are executing stage {current_stage} ({stage_name}) for {wrk_id}.

Context from completed stage {completed_stage}:
{context_summary}

Load these files before starting:
{entry_reads}

Stage contract: scripts/work-queue/stages/stage-{current_stage:02d}-*.yaml
Skills required: as listed in the stage contract.

When stage work is complete, run:
  uv run --no-project python scripts/work-queue/exit_stage.py {wrk_id} {current_stage}

Return a one-paragraph summary for the orchestrator log when done.
If exit_stage.py prints action=await_user_approval, stop and report the gate.

Chained stage groups: if checkpoint has chained_stages, the subagent prompt covers all stages in the group (e.g. stages 2/3/4 in one subagent).

Phase 3 — SKILL.md execution model update

work-queue/SKILL.md: document subagent-per-stage (or per chained-group) execution model. work-queue-workflow/SKILL.md: Stage Auto-Loop section + subagent prompt template.

Deferred

  • Retry logic if subagent exits non-zero
  • Parallel stage execution for independent stages
