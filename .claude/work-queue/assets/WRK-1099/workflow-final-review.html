1. --subcategory <name> filter

Add a --subcategory flag alongside the existing --category flag. Enables narrowing to e.g. --subcategory skills within the harness category, rather than seeing all 40+ harness items at once.

bash scripts/work-queue/whats-next.sh --subcategory skills
bash scripts/work-queue/whats-next.sh --category harness --subcategory work-queue

Implementation: read the subcategory: frontmatter field (already present) and apply an additional filter in process_file() when FILTER_SUBCATEGORY is set.

2. Workstation-aware recommendation sort

Within each section (HIGH PRIORITY, NEWLY UNBLOCKED, MEDIUM), items whose computer: field matches the current machine (hostname) are listed first; items on other machines follow, visually separated or labelled.

Current behaviour: all items interleaved regardless of machine — operator sees items they cannot run on the current workstation mixed in with items they can.

Desired behaviour:

★  HIGH PRIORITY
  [this machine: ace-linux-1]
  WRK-1097   work-queue   ace-linux-1   fix(work-queue): surface unclaimed…
  WRK-1098   skills       ace-linux-1   ecosystem terminology skill
  [other machines]
  WRK-125    marine       acma-ansys05  OrcaFlex module roadmap

Implementation: after collecting items per section, partition by computer == $(hostname) and render current-machine items first with a [this machine] sub-header, then other machines.

Why — - `/whats-next skill/harness` was attempted as a natural invocation — the flag didn't exist, causing confusion. `--subcategory skills` makes...
  • /whats-next skill/harness was attempted as a natural invocation — the flag didn't exist, causing confusion. --subcategory skills makes that intent expressible.
  • Operator on ace-linux-1 seeing acma-ansys05 items at the top of recommendations creates false leads — they can't run OrcaFlex jobs from their current seat.
  • Both changes are purely cosmetic/filtering; no data model changes required.
