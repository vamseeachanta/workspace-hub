The current 20-stage lifecycle runs as a single long conversation. This causes: 1. Context compaction — hard-stop instructions compressed out; agent forgets Stage 5 gate 2. Context compaction and context rot are structural, not behavioral. No amount of SKILL.md text fixes them — only isolation does. Each stage starting fresh from the previous stage's artifacts is the same principle used by CI/CD pipelines: each job gets a clean workspace built from the previous job's published artifacts, not from a shared mutable state.

This also enables: - Resumability: any stage can be resumed independently after a session drop - Parallel execution: Route C stages 6/8/9 can run concurrently if isolated - Engineering quality: heavy stages get their own full sub-skill chain, not a subset of a compressed context

Executive Summary

The 20-stage WRK lifecycle currently runs as a single long conversation, causing four structural failures: context compaction (hard-stop instructions disappear mid-session), context rot (earlier stage noise pollutes later stages), stage jumping (no mechanical barrier), and quality drift (heavy stages treated identically to light stages).

The fix is stage isolation: each stage starts fresh from the previous stage's exit artifacts — the same model as CI/CD job isolation. Autonomous stages run as Task agents (fresh context). Human-in-loop stages run in the current session (human IS the new context).

A single WRK-NNN-lifecycle.html file captures the entire WRK journey. Every stage exit updates it. You track progress in one place — no separate plan-draft.html, plan-final.html, or implementation.html files.

Delivery: P1 hookify backstops → P2 start/exit-stage.sh → P3 twenty YAML contracts + twenty micro-skills → P4 heavy-stage depth enforcement.

Stage status strip (top of every lifecycle HTML)

A compact visual bar at the top of the file shows all 20 stages at a glance — each as a numbered chip coloured by status: done in-progress pending blocked. Click to jump to that stage's section.

Update protocol

  • exit-stage.sh calls generate-html-review.py WRK-NNN --stage N --update after validating exit artifacts — the generator appends/updates the stage section in the existing file.
  • start-stage.sh (human_session only) flips the stage chip to in-progress before loading artifacts, so you can see the stage is open for review before any action is taken.
  • Push to origin after every update — lifecycle HTML is always current on the remote.

Every stage section in the lifecycle HTML uses the same field template. Autonomous stages fill agent name; human-in-loop stages require a real name. This makes gate checking uniform and positions the system for fully agentic execution.

WRK-1028 Cross-Review Package — Stage 6

Plan: Stage-isolated WRK lifecycle

Problem

The 20-stage WRK lifecycle runs as a single long conversation causing: 1. Context compaction — hard-stop instructions compressed away mid-session 2. Context rot — earlier stage noise pollutes later stages 3. Stage jumping — no mechanical barrier between stages 4. Quality drift — heavy stages treated same as light stages

Architecture Decision

Stage-isolation model: each stage starts fresh from previous stage's exit artifacts. - task_agent — autonomous stages spawn fresh Task agent (zero context rot) - human_session — gate stages (1,5,7,17) run in current session (human IS new context) - chained_agent — light stage groups in one Task agent (2+3+4, 8+9)

Single Lifecycle HTML

One WRK-NNN-lifecycle.html per WRK. Updated at every stage exit. Every stage has: reviewed_by, confirmed_by, decision, notes schema. No separate plan-draft/implementation HTML files.

Stage Classification

  • Light (4KB budget): 1, 3, 8, 9, 18, 19, 20 — work-queue-workflow only
  • Medium (8KB): 2, 4, 6, 11, 13, 14, 15, 16 — work-queue-workflow + domain skill
  • Heavy (16KB): 5, 7, 10, 12, 17 — + brainstorming/TDD/systematic-debugging/workflow-html
