feat(session): Interactive Session Analysis

What

Enhance /session-start with an interactive analysis phase that assesses — at session open — whether the key rules, skills, and context files are loaded and acknowledged. If proactive hook-based session logging has not fired, trigger it interactively.

This replaces the current "surface and move on" briefing with a structured load assessment → gap surfacing → user confirmation → log loop.

Why

Agents (Claude, Codex, Gemini) drift from rules because they rely on training knowledge rather than reading the canonical rule files. A session-start inventory that confirms each critical rule is in scope — and logs that confirmation — creates an audit trail and forces the agent to load the file before starting work.

Examples surfaced this session (2026-03-03): - python-runtime.md rule: agents used python3 / .venv/bin/python instead of uv run - File-structure skills: agents placed files without consulting file-taxonomy or repo-structure — because neither was explicitly loaded

Scope

1. Session Load Assessment (new phase in /session-start)

At session open, check and report:

Category Check Pass Condition
Runtime rule .claude/rules/python-runtime.md loaded File read + key phrase "uv run" acknowledged
Git rule .claude/rules/git-workflow.md loaded File in context
File placement file-taxonomy/SKILL.md version current version ≥ 1.6.0
Skill map .claude/docs/file-structure-skills-map.md accessible File exists
Memory MEMORY.md < 200 lines wc -l check
Active WRK .claude/state/active-wrk set or empty Surfaced to user
Quota config/ai-tools/agent-quota-latest.json fresh modified ≤ 7 days ago

2. Interactive Gap Resolution

For each CHECK that fails, prompt the user:

⚠ python-runtime.md — not loaded this session
  → Read it now? [y/n]  (y = read + log, n = skip with warning)

Do not block session startup on gaps — surface and let user decide.

3. Session Logging (if hooks have not fired)

Check whether logs/orchestrator/claude/session_YYYYMMDD.jsonl has entries from the last 60 seconds (i.e., hook fired at session open).

If not:

# Write a session-open marker manually
echo '{"event":"session_open","ts":"<ISO>","machine":"<hostname>","wrk":"<active-wrk>"}' \
  >> logs/orchestrator/claude/session_$(date +%Y%m%d).jsonl

Log the load assessment result too:

{"event":"load_assessment","ts":"<ISO>","rules_loaded":["python-runtime","git-workflow"],
 "rules_missing":[],"active_wrk":"WRK-NNN","quota_fresh":true}

4. Skill: /session-interactive-analysis

New standalone skill (can also be called mid-session after /clear): - File: .claude/skills/workspace-hub/session-interactive-analysis/SKILL.md - Runs the load assessment + gap resolution loop without the full session-start briefing - Useful when returning from context wipe or switching WRK scope mid-session

5. Session Learnings Integration

After each session end, comprehensive-learning Phase 1 reads the load assessment log entries and: - Tracks which rules were not loaded most often → promotes those rules to "always pre-load" list in python-runtime.md / git-workflow.md ## Session Learnings sections - Flags skills that were frequently consulted but not listed in load assessment → adds them to the next session's check table

Design Decision (2026-03-06): Option C — Split Prevention + Analytics

Session start (non-interactive): Auto-read 3 drift-risk files into context — no prompts, no checks, < 2s. Post-session (comprehensive-learning Phase 1): Scan logs for actual violations → trend tracking.

Drift-risk files (all 3): - .claude/rules/python-runtime.md — prevents bare python3 (silent failure) - .claude/rules/git-workflow.md — prevents non-conventional commits - .claude/skills/workspace-hub/file-taxonomy/SKILL.md — prevents wrong file placement
