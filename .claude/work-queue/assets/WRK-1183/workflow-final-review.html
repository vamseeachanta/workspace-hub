The skill should define a repeatable workflow:

  1. Identify — locate Excel files containing engineering calculations (from past projects, coursework, or reference material)
  2. Extract — pull out the generic methodology: formulas, input ranges, expected outputs, validation cases, unit systems, applicable standards
  3. Sanitize — run legal-sanity-scan to ensure zero client identifiers, project names, or proprietary data remain
  4. Archive — save as structured YAML in knowledge/dark-intelligence/<category>/<subcategory>/:
  5. equations (LaTeX)
  6. input/output schemas
  7. worked examples with known answers
  8. standard references
  9. methodology notes
  10. Generate TDD test data — convert worked examples to pytest test cases with concrete assertions (known inputs → known outputs)
  11. Implement — use the extracted methodology + test data to build clean implementations in the 4 public repos (digitalmodel, worldenergydata, assethold, assetutilities)
  12. Produce calc report — generate calculation-report YAML per the calculation-report skill
Why — - Legacy Excel files contain verified calculations from real projects — years of engineering knowledge - Carrying original Excel files creat...
  • Legacy Excel files contain verified calculations from real projects — years of engineering knowledge
  • Carrying original Excel files creates legal/IP risk (client data, project names, proprietary formats)
  • Extracting only the methodology strips all risk while preserving 100% of the engineering value
  • Worked examples from Excel become TDD test assertions — more reliable than hand-derived values
  • University coursework Excel files are also a rich source (no legal risk but same extraction benefit)
  • This compounds: every extracted methodology makes the next calculation implementation faster
