Currently, Claude Code sessions on all workstations run with --dangerously-skip-permissions, which bypasses all tool-call approval checks globally. This is a blunt instrument — it hides exactly which permissions are actually needed.

Desired end-state: - Audit all commands used across sessions (Bash tool, tool patterns, MCP calls) to derive the minimal required permission set. - Define a structured allow/deny list in .claude/settings.json that encodes exactly those permissions. - File travels with the repo — any new workstation clone gets the right permissions automatically, without needing --dangerously-skip-permissions. - Deny list blocks known-dangerous patterns (e.g., rm -rf /, chmod 777, sudo *) already identified in the settings audit (WRK-1016).

Why — - Security posture: least-privilege principle; dangerous operations require explicit human approval rather than being silently allowed. - Po...
  • Security posture: least-privilege principle; dangerous operations require explicit human approval rather than being silently allowed.
  • Portability: permissions are version-controlled and consistent across ace-linux-1, ace-linux-2, acma-ansys05, and any future workstations.
  • Auditability: the allow list is a living document of what the ecosystem actually needs.
Acceptance Criteria — 1. Audit script identifies all Bash/tool patterns used in recent sessions and outputs a candidate allow list. 2. `.claude/settings.json` upd...
  1. Audit script identifies all Bash/tool patterns used in recent sessions and outputs a candidate allow list.
  2. .claude/settings.json updated with allow and deny sections derived from the audit; committed to workspace-hub so it travels with the repo.
  3. --dangerously-skip-permissions removed from session launch commands on ace-linux-1; sessions function correctly with defined permissions only.
  4. Deny list covers at minimum: rm -rf /, chmod 777, git push --force to main, and patterns from .claude/rules/security.md.
  5. Documentation in .claude/docs/ explains how to extend the allow list when new legitimate commands are needed.
  6. Tested: fresh session on ace-linux-1 works without --dangerously-skip-permissions.
  7. TDD: ≥3 tests validating deny-list enforcement and allow-list coverage.
