### Verdict: REQUEST_CHANGES

### Summary
The plan's core premise — "Excel cell values ARE the test data" via openpyxl data_only=True — has a critical silent-failure path: if any XLSX was saved without recalculation (common for programmatically-generated files), data_only=True returns None for all formula cells, producing test assertions against None/empty values. This undermines the entire POC's correctness guarantee. Additionally, the VBA extraction from untrusted files needs an explicit safety boundary documented in the plan.

### Issues Found
- [P1] Critical: specs/modules/wrk-1247-xlsx-formula-extraction-poc.md:36-38 — The two-pass strategy assumes data_only=True always returns cached computed values. openpyxl's data_only=True only returns the LAST CACHED value stored in the XML; if a spreadsheet was saved without recalculation (common for programmatically-generated XLSX, LibreOffice exports, or files saved with 'manual calculation' mode), ALL formula cells return None. The plan MUST add a pre-validation step after Pass 1 that counts how many formula cells returned None vs actual values, and abort/warn when the cache-miss rate exceeds a threshold. Without this, the POC will silently generate pytest assertions like `assert result == None` — tests that pass trivially but verify nothing.
- [P1] Critical: specs/modules/wrk-1247-xlsx-formula-extraction-poc.md:109-116 — VBA extraction via oletools processes potentially malicious macro content from untrusted .xlsm files. While oletools is a security analysis tool and doesn't execute VBA, the plan's Step 5 mentions 'Parse VBA function signatures → Python def stubs'. The plan must explicitly state: (1) VBA source is treated as untrusted string data only — never passed to exec/eval/compile, (2) generated Python stubs are signature-only scaffolds with `raise NotImplementedError`, not translated logic, (3) oletools VBA_Parser should be wrapped in a try/except that catches oletools-specific exceptions (corrupted OLE streams can raise unexpected errors). Document this safety boundary in the plan.
- [P1] Blocker: .claude/work-queue/pending/WRK-1247.md:28-29 — The WRK references 'ace_project + dde_project' as source file locations for POC file selection. Per legal-compliance.md, client project names/codenames must NEVER appear in code. The select_poc_xlsx.py script and poc-file-list.yaml must use sanitized/generic path references (e.g., environment variables or config-based path mapping). Ensure these identifiers do not leak into committed code, YAML artifacts, or the selection script's source.

### Suggestions
- Add a cache-hit validation step between Pass 1 and Pass 2: count formula cells where data_only=True returned None. If >50% of formula cells have None values, log a warning and either (a) use the `formulas` library's ExcelModel.calculate() as fallback to compute values, or (b) flag the file as 'uncalculated' in the yield report and exclude it from test generation. This is the difference between a POC that proves the concept vs one that silently produces vacuous tests.
- Add an explicit 'Safety Boundaries' section to the plan documenting: VBA strings are never evaluated, Python stubs raise NotImplementedError, and all file paths from project sources are sanitized through the legal deny-list before appearing in any committed artifact.
- For the FormulaXlsxParser.can_handle(), ensure it accepts both .xlsx and .xlsm extensions — the existing XlsxParser only checks .xlsx (line 29), and the plan doesn't explicitly show the new parser's extension check including .xlsm.

### Questions for Author
- What percentage of the 23K XLSX files are expected to be programmatically-generated (e.g., exported from databases, engineering tools) vs manually created in Excel? This determines how severe the None-cached-value problem will be at scale.
- Are 'ace_project' and 'dde_project' client project identifiers that appear in the legal deny list, or are they internal workspace directory names? If the former, the selection script needs a path-aliasing layer before any code is written.
