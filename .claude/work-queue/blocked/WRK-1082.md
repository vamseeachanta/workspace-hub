## Mission
Ensure every tier-1 repo has a committed, current uv.lock so builds are
reproducible and dependencies with known CVEs are surfaced before they reach
engineering outputs.

## What
- Audit script: `scripts/quality/lockfile-audit.sh`
  - For each tier-1 repo: check `uv.lock` exists and is committed
  - Run `uv lock --check` to verify lock is consistent with pyproject.toml
  - Warn if lock file not updated in >30 days
  - Run `uv audit` (OSV advisory database) for known CVEs in pinned deps
  - Output: per-repo table (lock present | consistent | age | CVE count)
- Fix inconsistent locks: run `uv lock` in affected repos, commit
- `uv audit` output: `config/quality/vuln-report-YYYYMMDD.json`; CRITICAL CVEs
  block CI gate (WRK-1064 integration)
- Weekly cron: add lockfile-audit.sh at Sunday 04:30

## Why
Lock file drift between machines is one of the hardest bugs to diagnose — identical
code, different behaviour, because dep resolution diverged. `uv audit` closes a
security gap: pinned deps can still contain published CVEs. Both are invisible without
tooling.

## Plan

1. **Add OGManufacturing to `scripts/readiness/harness-config.yaml`** tier1_repos; add
   `repo_paths` mapping (name → relative path from workspace root).
2. **Write `scripts/quality/lockfile-audit.sh`** — read-only audit script; reads repo list +
   paths from harness-config.yaml; for each repo:
   - Check `uv.lock` filesystem presence AND git-tracked state
   - `uv lock --check` → consistent with pyproject.toml?
   - Compute age from `git log` → warn if >30 days
   - Skip with reason if: no pyproject.toml (`no_pyproject`), no uv.lock (`no_lockfile`),
     export fails (`export_failed`)
   - CVE scan: `uv export --frozen --no-dev -o /tmp/wrk-audit-<repo>.txt` then
     `uv tool run pip-audit -r /tmp/... --format json` (audits pinned lockfile deps)
   - `high` severity → warn; `critical` → fail; both affect `ci_gate_recommendation`
   - Collect ALL repos first, then set exit code; never abort early
   - Print table: repo | repo_status | lock_present | lock_tracked | consistent | age | critical | high
   - Write `reports/quality/vuln-report-YYYYMMDD-HHMMSS.json` (gitignored dir, timestamped);
     also write `reports/quality/vuln-report-latest.json` symlink for CI consumption
3. Run the script → capture vuln-report JSON; inspect output.
   (Lock remediation — `uv lock` + commit — is a separate manual step if needed.)
4. **Add weekly cron** at Sunday 04:45 to `scripts/cron/setup-cron.sh` ENTRIES array
   (canonical installer); also mirror in crontab-template.sh for documentation;
   re-run `setup-cron.sh` to install on dev-primary.
5. **Codex cross-review** of the script; fix any MAJOR findings.
6. **Commit all artifacts** — harness-config.yaml update, lockfile-audit.sh,
   setup-cron.sh + crontab-template.sh (no vuln-report in git — gitignored).
