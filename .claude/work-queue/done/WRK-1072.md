## Mission
Give tier-1 engineering repos versioned releases with changelogs so it's always
traceable which version of a calculation module produced a given report or result.

## What
- Versioning standard: SemVer (`MAJOR.MINOR.PATCH`) for all 5 tier-1 repos
- Release script: `scripts/release/cut-release.sh <repo> <version> [--dry-run]`
  - Bumps version in `pyproject.toml`
  - Generates CHANGELOG entry from conventional commits since last tag
    (`git log --pretty=format:"%s" <last-tag>..HEAD | conventional-changelog`)
  - Creates git tag `v<version>` in the repo
  - Updates hub-level `config/releases/release-manifest.yaml` with all repo versions
- `release-manifest.yaml`: single source of truth for current version of every
  tier-1 repo — agents read this to know "what code is active"
- Changelog format: `CHANGELOG.md` per repo, Conventional Commits grouped by type
- Trigger: manual only (not automated); run before any report that should be reproducible

## Why
Engineering reports reference modules like "CP design per assetutilities." Without
versioning, there is no way to reproduce a historic calculation or verify that the
correct version was used. Semantic versioning closes this traceability gap and is
standard practice for engineering software.

## Acceptance Criteria
- [ ] `pyproject.toml` has `version` field in all 5 tier-1 repos
- [ ] `cut-release.sh` bumps version, generates CHANGELOG, tags repo
- [ ] `CHANGELOG.md` generated from conventional commits per repo
- [ ] `release-manifest.yaml` updated on every release
- [ ] `--dry-run` flag shows what would be done without writing
- [ ] Cross-review (Codex) passes
