## Plan (v2 — post cross-review)

### Execution boundary (design decision)
CI runs **repo-local**: each tier-1 repo has its own docs workflow job calling
`uv run mkdocs build` inside that repo. `build-api-docs.sh` is the **hub orchestrator**
for local multi-repo builds — it shells into each repo dir, installs deps, builds.
Single source of truth for local dev; CI mirrors it per repo.

### Phase 0 — Pilot: assetutilities end-to-end (before any hub script)
1. Add `mkdocs` + `mkdocstrings[python]` to assetutilities dev deps (`uv add --dev`)
2. Write `mkdocs.yml` at repo root with explicit mkdocstrings config:
   - `handlers.python.options.show_source: false`, `docstring_style: google`
   - `handlers.python.options.ignore_init_summary: true`
   - `import_mode: importlib` to avoid side-effect imports
3. Write `docs/api/index.md` entry point with `::: assetutilities` reference
4. Run `uv run mkdocs build --strict` — fix all errors until clean
5. Verify `site/` created, API nav visible, `site/` in `.gitignore`
6. TDD: write `tests/docs/test_build_api_docs.sh` against this repo first
7. **Gate**: assetutilities pilot PASS before Phase 1

### Phase 1 — Hub orchestrator script + manifest
1. `scripts/docs/build-api-docs.sh [--repo <name>] [--serve] [--strict]`
   - Skips repos where `<repo>/mkdocs.yml` absent (no-op success, logs "SKIP: no mkdocs.yml")
   - `uv sync --dev` first; then `uv run mkdocs build`; `--strict` flag optional
   - `--serve`: starts `mkdocs serve` (one repo only; binds localhost:8000; user Ctrl+C)
   - `--repo <name>` without mkdocs.yml = skip + warn (not an error)
   - Colored output, per-repo exit codes; fails fast on non-zero
2. `config/docs/docs-manifest.yaml` — **derived artifact**, not hand-maintained.
   `build-api-docs.sh` regenerates it on each run from repo filesystem inspection.
   Manual edits will be overwritten. Structure: `repo → {enabled, site_path, serve_url}`
3. Extend `tests/docs/test_build_api_docs.sh` for hub orchestration
4. `--serve` test: spawn server, poll `localhost:8000` up to 15s for HTTP 200, then SIGTERM,
   assert clean shutdown (exit 0 or 130). No fixed sleep.

### Phase 2 — Roll out mkdocs.yml to remaining 4 repos
For worldenergydata, digitalmodel, assethold, ogmanufacturing:
- Same mkdocs.yml + docs/api/index.md + dep pattern from Phase 0
- digitalmodel: build with `--no-strict` first; measure build time; enable `--strict`
  only after existing docstring issues are addressed (non-blocking for this WRK)
- Rollback policy: if a repo cannot build cleanly, add `docs_build: disabled` flag
  in docs-manifest.yaml and skip it in hub script — main branch unblocked

### Phase 3 — CI integration (standardized across all repos)
- One reusable pattern: `docs.yml` workflow in each repo (new or extended)
- Job: `cd <repo> && uv sync --dev && uv run mkdocs build --strict`
- Trigger: push to main; also run as advisory on PRs
- digitalmodel: `--no-strict` until docstring quality improves; add comment
- CI does NOT publish/deploy (local-only for now; WRK-future for gh-pages)

### Test Strategy
- Shell contract tests (fast, mock mkdocs): flags, exit codes, cwd handling
- Integration — Tier 1: `assetutilities` full `mkdocs build`; assert site/ + nav entry
- Integration — Tier 2: `digitalmodel` smoke build (1 module only) to validate import handling
- `--serve` test: assert mkdocs serve process starts and is killed cleanly (no long-running)
- AC matrix: each AC below maps to ≥1 test case

### Risks
- R1 (Sphinx coexistence): handled — `site/` gitignored, separate from `docs/`
- R2 (digitalmodel build time): measure in Phase 2; `--no-strict` + CI timeout=10m guard
- R3 (griffe import side-effects): mitigated by `import_mode: importlib` setting
- R4 (existing docstring gaps block CI): mitigated by `--no-strict` + rollback flag

## Mission
Auto-generate browsable API reference docs from docstrings so engineers can navigate
module interfaces without reading source code — making engineering WRK exploration fast.

## What
- Toolchain: MkDocs + mkdocstrings (Google-style docstring parser) per repo
- Config: `mkdocs.yml` in each tier-1 repo root; `docs/api/` auto-generated from src/
- Generation script: `scripts/docs/build-api-docs.sh [--repo <name>] [--serve]`
  - Runs `mkdocs build` per repo; outputs to `site/` (gitignored)
  - `--serve` flag launches `mkdocs serve` for live local browsing
- CI integration: build docs on every push to main; fail build if any autodoc
  import error occurs (catches broken docstrings early)
- Hub-level index: `config/docs/docs-manifest.yaml` — repo → local doc URL map;
  `/work status WRK-NNN` can link to relevant module docs

## Why
WRK-1058 will enforce docstring completeness. WRK-1059 will enforce reference doc
accuracy. Without a rendering pipeline those docstrings stay in source — useful but
not browsable. MkDocs + mkdocstrings converts them to searchable HTML in one command.

## Acceptance Criteria
- [ ] `mkdocs.yml` + mkdocstrings config in all 5 tier-1 repos
- [ ] `build-api-docs.sh` builds all repos without errors (assetutilities strict; others advisory)
- [ ] API docs appear in site nav (not just generated — discoverable by engineers)
- [ ] `--serve` starts a live local preview server for specified repo
- [ ] `build-api-docs.sh` exits non-zero when mkdocs reports a collection/import error
- [ ] `site/` dir created at `<repo>/site/` with deterministic output; added to `.gitignore`
- [ ] `docs-manifest.yaml` at hub level mapping repo → local site path + serve URL
- [ ] CI docs job present in all 5 repos (standard pattern); runs on push to main
- [ ] Repos that cannot build cleanly have `docs_build: disabled` in manifest + skip in script
