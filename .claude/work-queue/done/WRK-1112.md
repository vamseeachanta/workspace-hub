# feat(review): Limit Cross-Review Iterations to 3 — Announce Budget Upfront

## Mission

Cap all AI code-assistant cross-review and review cycles at **3 iterations maximum**
and surface that constraint in the review prompt so the reviewer plans its highest-value
comments first.

## Why

Unbounded review loops waste quota, slow delivery, and dilute feedback quality — reviewers
iterate indefinitely rather than prioritising the most impactful comments. A hard cap of 3
forces the reviewer to front-load critical findings and prevents diminishing-return churn.

## What

### Phase 1 — Enforce the 3-iteration cap in cross-review scripts

- `scripts/review/cross-review.sh` — add `MAX_REVIEW_ITERATIONS=3` guard; abort with
  an explanatory message if a caller tries a 4th pass on the same file/WRK.
- `scripts/review/submit-to-codex.sh` and `scripts/review/submit-to-gemini.sh` — track
  invocation count per WRK (persisted to `assets/WRK-NNN/review-iteration.yaml`);
  refuse iteration 4+.

### Phase 2 — Announce the budget in the reviewer system prompt

Prefix every cross-review submission with a standard preamble:

```
You are reviewing [WRK-NNN] — iteration N of 3 (maximum).
Plan your feedback to maximise impact within this budget:
  • Iteration 1: blockers and security issues only
  • Iteration 2: major design / correctness issues
  • Iteration 3: minor / style / nice-to-haves
After iteration 3 no further review passes will be accepted.
```

Wire the preamble into `cross-review.sh` and both `submit-to-*.sh` wrappers.

### Phase 3 — Update skill documentation

- `work-queue/SKILL.md` §Cross-Review: note the 3-iteration cap.
- `work-queue-workflow/SKILL.md` Stage 13 contract: add iteration-count gate.
- `workflow-gatepass/SKILL.md`: add R-28 — "cross-review iteration count ≤ 3".

## Acceptance Criteria

- [ ] `review-iteration.yaml` created per-WRK on first review; incremented each pass
- [ ] `cross-review.sh` exits non-zero with clear message on iteration 4+
- [ ] Both `submit-to-*.sh` scripts enforce the same cap
- [ ] Preamble with iteration count and budget guidance injected into every review prompt
- [ ] SKILL.md files updated with the 3-iteration policy
- [ ] TDD: ≥5 tests covering cap enforcement, yaml persistence, preamble injection

## TDD Approach

Write tests before implementation:
1. `test_review_iteration_yaml_created_on_first_pass`
2. `test_review_iteration_increments_correctly`
3. `test_fourth_iteration_blocked_with_error`
4. `test_preamble_includes_iteration_number_and_budget`
5. `test_cap_applies_to_codex_and_gemini_wrappers`

## Non-goals

- Changing the content of review criteria themselves
- Limiting human reviewer iterations
- Retroactively enforcing the cap on already-closed WRKs
