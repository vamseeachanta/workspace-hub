## Mission

Clean up non-actionable error output from work-queue scripts so the CLI shows only meaningful information during `/work run`.

## What

Noisy stderr sources identified:

1. **exit_stage.py:55** — `WARN: D-item checks unavailable (ImportError)` prints on every stage exit even when irrelevant
2. **stage_dispatch.py:75,78** — `D-ITEM WARN:` / `D-ITEM BLOCKED:` messages lack severity clarity
3. **archive-item.sh:147** — `2>/dev/null` on `generate-html-review.py` hides real errors silently
4. **claim-item.sh:125** — `Quota file missing` warning prints to stdout, mixing with normal output
5. **comprehensive-learning-nightly.sh:84** — `|| true` swallows real failures from detect-drift.sh
6. **check-gates-green.sh:22** — `2>&1` mixes Python tracebacks with gate status output

## Why

Non-actionable warnings clutter the terminal during normal `/work run` sessions. Users cannot distinguish real errors from expected noise, reducing trust in the workflow output.

## Plan

- Fix stderr routing in 4 scripts: exit_stage.py, stage_dispatch.py, archive-item.sh, claim-item.sh
- Add severity prefixes to distinguish [WARN] from [ERROR]
- Redirect suppressed output to log files instead of /dev/null
- Write TDD tests for each fix
- Descope items 5-6 (comprehensive-learning, check-gates-green) if investigation needed

## Acceptance Criteria

- [x] `exit_stage.py` D-item warning suppressed when no D-items exist (silent return on ImportError)
- [x] `stage_dispatch.py` warnings prefixed with severity: `[WARN]` vs `[ERROR]`
- [x] `archive-item.sh` logs HTML generation errors to `html-gen.log` instead of `/dev/null`
- [x] `claim-item.sh` quota warning goes to stderr (not stdout)
- [x] No visible warnings during a clean `/work run` of a simple WRK item
