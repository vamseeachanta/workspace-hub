## Mission

Replace the current ad-hoc HTML generation with exactly **two canonical HTML files** per WRK
item — `plan.html` and `lifecycle.html` — regenerated at every stage exit, so users only
ever need to refresh those two URLs.

## What

Currently, `generate-html-review.py` can emit separate plan-draft / plan-final / lifecycle
files at different stages, and not all stages regenerate every file. This leads to stale
views and unclear "which file to open" UX.

Desired end-state:

### `plan.html`
- **Top section**: latest approved plan (always kept current).
- **Stage-change log**: a collapsible per-stage section showing what changed in the plan at
  that stage (diff-like narrative or structured table).
- **Stage-progress ring**: same coloured stage-circles widget used in `lifecycle.html`
  so the user can see how far the item has progressed while reading the plan.
- Regenerated on every stage exit (stages 1–20).

### `lifecycle.html`
- Unchanged widget + stage circles, but also regenerated on every stage exit (currently
  regenerated only on some stages).

### Skill integration
- `workflow-html/SKILL.md` and `workflow-gatepass/SKILL.md` should reference these two
  canonical files.
- `exit_stage.py` should call `generate-html-review.py WRK-NNN --plan` and
  `generate-html-review.py WRK-NNN --lifecycle` after every successful stage exit.

## Why

- Single refresh URL per document type reduces cognitive overhead.
- Plan evolution is visible inline — no need to diff separate files.
- Stage-circle widget in both HTMLs gives immediate orientation.

## Acceptance Criteria

1. `plan.html` exists in `assets/WRK-NNN/` for any item that has completed stage 4b.
2. `plan.html` top section always shows the most recent approved plan text.
3. `plan.html` includes per-stage change log (at minimum: stage number, summary of what
   changed; ideally: diff of plan bullet points).
4. `plan.html` includes the same stage-circle progress widget as `lifecycle.html`.
5. `lifecycle.html` is regenerated on every stage exit (no gaps).
6. `exit_stage.py` (or equivalent hook) triggers both regenerations automatically.
7. Both files use `<meta http-equiv="refresh" content="30">` for auto-refresh.
8. Existing `workflow-html/SKILL.md` updated to document the two-file contract.
9. TDD: ≥3 passing unit/integration tests covering HTML generation and stage-exit hook.
10. Stage 12 renderer shows `ac-test-matrix.md` as a PASS/FAIL table inline (not just gate existence).
11. Stage 10 renderer shows `execute.yaml → integrated_repo_tests` as a per-test name/command/result table.
12. Stage 10 renderer shows `execute.yaml → changes[]` as a bullet list of what was modified.
13. Stage 14 gate table includes the `details` column from `gate-evidence-summary.json` (not just gate name + status).
14. Stage 13 uses the same cross-review renderer as Stage 6 (verdict + findings with severity).
15. Each stage section header shows duration (start→end) from `stage-evidence.yaml` where available.
