# Skill capability assessment for WRK-624 workflow governance skill set

## What

Run structured capability assessments for each skill that is part of the WRK-624 workflow
governance harness. Produce a per-skill scorecard that measures: (a) whether the skill does
what it claims, (b) whether the bare model can do the same without the skill, and (c) whether
any two skills overlap enough to merge or retire one.

## Phase 0: Skill Knowledge Map (prerequisite to evals)

Before running capability assessments, build a knowledge map of the 8 skills showing how
they relate, flow into each other, and where their boundaries sit. This map makes the
skill-vs-skill overlap analysis (Dimension 3) grounded rather than speculative.

**What to capture per skill:**
- **Triggers**: what user phrases / contexts load this skill
- **Inputs**: what state/context does the skill expect to already exist
- **Outputs**: what artifacts, decisions, or state changes does it produce
- **Handoffs**: which skills does it explicitly reference or hand off to next
- **Scope boundary**: what does it explicitly NOT do (negative scope)

**Advantages of the knowledge map:**
- Exposes implicit dependencies between skills (e.g., `session-start` → `work-queue` → `workflow-gatepass`)
- Reveals scope gaps — workflows that cross skill boundaries with no owner
- Identifies redundant coverage — two skills claiming the same trigger phrase
- Provides the DAG needed to sequence evals correctly (upstream skills first)
- Becomes a living reference doc for onboarding new sessions to the workflow harness

**Deliverable**: `specs/skills/skill-knowledge-map.md` — DAG diagram + per-skill boundary table

**Format suggestion:**
```
session-start → work-queue (capture) → work-queue-workflow (lifecycle) → workflow-gatepass (gates)
                                     → resource-intelligence (context)  ↗
                                                                wrk-lifecycle-testpack (evidence)
comprehensive-learning ← (runs at session end, reads all WRK artifacts)
cross-review ← (called inline during gatepass stages 8/13)
```

## Eval dimensions (per skill)

### 1. Capability eval
- Define the 3-5 core behaviors the skill claims to provide
- Run a representative task with skill injected → record outputs
- Score: Does each claimed behavior manifest? (pass / partial / fail)

### 2. With vs without (A/B — bare model)
- Run the same representative task **without** the skill (bare model)
- Compare outputs on same scoring rubric
- Delta score = skill contribution; near-zero delta = retirement candidate

### 3. Skill vs skill (A/B — closely related)
Pairs to evaluate for overlap:
- `work-queue` vs `work-queue-workflow` — do both cover lifecycle routing?
- `workflow-gatepass` vs `wrk-lifecycle-testpack` — both test gate compliance; overlap?
- `session-start` vs `work-queue` capture phase — redundant session-init steps?
- `comprehensive-learning` vs `improve` — scope boundary clear?

### 4. Procedural eval
- For skills with defined step sequences (session-start, workflow-gatepass), verify the
  model follows the prescribed stage order when the skill is injected
- Score: stages followed / total required stages
