# WRK-1026 — work-queue-workflow Stage 5: pseudocode + tests/evals in planning

## Mission
**Enrich Stage 5 planning contract and define single lifecycle HTML model across `work-queue-workflow/SKILL.md` and `workflow-html/SKILL.md`.**

Scope: `work-queue-workflow/SKILL.md` + `workflow-html/SKILL.md`. Not in scope: `work-queue/SKILL.md`, `workflow-gatepass/SKILL.md`, or `generate-html-review.py` code changes (captured as WRK-1027).

## What
Stage 5 of the work-queue-workflow skill currently produces a plan artifact but does not explicitly require:
1. Pseudocode for non-trivial logic steps in the plan
2. A tests/evals section specifying how the implementation will be verified

Add both as mandatory elements of the Stage 5 plan draft output so that agents produce richer, more testable plans before user review.

## Why
Plans without pseudocode often result in implementation drift — the agent interprets ambiguous steps differently than intended. Plans without tests/evals leave the acceptance criteria vague, making Stage 12 (TDD/Eval) harder to execute. Adding these two elements at Stage 5 aligns with TDD-first principles and reduces rework.

## Acceptance Criteria
- [ ] Stage 5 section explicitly splits Route A (human-in-loop only, no scripts) vs Route B/C (human-in-loop + 3-agent independent planning after markdown saved)
- [ ] Route B/C post-save flow: all 3 agents draft independently → synthesis before Stage 6
- [ ] Stage 5 includes pseudocode requirement: function-level, objectives-first, N/A+reason allowed
- [ ] Stage 5 includes tests/evals subsection: ≥3 entries, N/A+reason allowed for pure-doc WRKs
- [ ] Stage 5 includes "Key Planning Skills" block (brainstorming, resource-intelligence, TDD, workflow-html)
- [ ] Stage 10 includes "Key Execution Skills" block (file-taxonomy, coding-style, systematic-debugging, TDD)
- [ ] Stage 5 exit checklist reflects all new items (≥8 total)
- [ ] Language is imperative/blocking (consistent with WRK-1017 Stage 5 hardening style)

## Agentic AI Horizon
- Pseudocode + tests/evals at plan time = machine-readable specification that future agents can directly execute; investing now builds the scaffold for fully autonomous WRK execution
- Shape it as structured YAML-embeddable pseudocode so a future eval harness can parse and auto-generate test stubs
