# feat(harness): add per-provider model/effort/knobs summary to daily cron checks

## Mission
**Extend `scripts/productivity/sections/ai-usage-summary.sh` so the daily cron report surfaces the active model, effort/thinking level, and key configuration knobs for each AI provider (Claude, Codex, Gemini), giving a quick at-a-glance config state alongside usage stats.**

## What
The daily `daily_today.sh` cron already shows AI usage stats via `ai-usage-summary.sh`. This WRK adds a per-provider configuration summary block that reads active settings (model name, effort/thinking level, temperature, context window, or other relevant knobs) from each provider's config and prints them in the daily report.

Example output format:
```
=== AI Provider Config ===
claude  : model=claude-sonnet-4-6  effort=medium  thinking=off
codex   : model=o4-mini             effort=medium  reasoning=medium
gemini  : model=gemini-2.5-pro      effort=standard
```

Sources to read from (investigate and confirm during execution):
- Claude: `~/.claude/settings.json` or `config/ai-tools/` — model + effort fields
- Codex: `~/.codex/config.json` or `CODEX_MODEL` env — model + reasoning effort
- Gemini: `~/.gemini/settings.json` or similar — model name

## Why
When switching between model tiers (Sonnet vs Opus, o4-mini vs o3, etc.) it is easy to forget which config is active. The daily report is the natural place to verify this at a glance — same session where quota is checked.

## Acceptance Criteria
- [ ] Identify config file/env var source for model + effort/knobs for each provider (Claude, Codex, Gemini)
- [ ] Add a "AI Provider Config" block to `ai-usage-summary.sh` that reads and prints the active config
- [ ] Gracefully handle missing config (print `not configured` rather than erroring)
- [ ] Output is concise — fits in 3-5 lines, one per provider
- [ ] Existing usage stats output is unchanged
- [ ] Manual test: run `bash scripts/productivity/sections/ai-usage-summary.sh` and confirm new block appears

## Agentic AI Horizon
- As model tiers proliferate (Haiku/Sonnet/Opus, mini/full, thinking on/off), config visibility in the daily report becomes increasingly valuable for cost/capability tradeoffs.
- Future: auto-flag if an expensive model (Opus, o3) is set as default when lighter model suffices.
