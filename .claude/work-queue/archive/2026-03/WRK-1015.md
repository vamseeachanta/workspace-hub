# feat(work-queue): category grouping and priority-sorted view for INDEX.md and /work list

## What

Add a `category:` field to WRK items and extend INDEX.md + `/work list` to group items by
category, sorted by priority within each group. The goal is a single-terminal session view
where the user can scan one category at a time, pick the highest-priority unblocked item,
and work through it — maximising focused productivity per ecosystem area.

## Why

The work queue currently has 400+ items. Without grouping, reviewing what needs attention
in any given area (e.g. harness, engineering domains, skills) requires scrolling through
the full list. Category grouping enables:

- **Targeted sessions**: open one category, work top-to-bottom by priority
- **Gap detection**: see which categories are overloaded vs underserved
- **Skill/WRK alignment**: the knowledge map (WRK-1010) shows skill flow; categories show
  WRK flow — together they give a full picture of where to invest effort

## Category + Sub-category Taxonomy

Two-level hierarchy: `category` (fixed set, 7 values) + `subcategory` (adaptive — grows as
work evolves, no fixed allowlist). Both fields stored in WRK frontmatter.

**Subcategory design principle**: subcategories are *discovered*, not prescribed. At capture
time, `infer-category.py` scans existing WRK values and suggests the closest match. If none
fits, a new subcategory name is minted — the only constraint is kebab-case. This keeps the
taxonomy honest to the actual work rather than an upfront guess about what the work will be.

### Category values (fixed)

| Category | Scope |
|----------|-------|
| `harness` | Workflow infrastructure, skills, queue, hooks, session |
| `engineering` | Domain implementations across all engineering repos |
| `data` | Data sources, document intelligence, standards |
| `platform` | Workstations, AI tools, plugins, cron |
| `business` | Finance, GTM, website, brochures |
| `maintenance` | Cleanup, refactors, test debt |
| `personal` | Non-code items |

### Seed subcategories (starting point — extend freely)

**harness**: `skills`, `work-queue`, `workflow`, `hooks`, `session`

**engineering**: `pipeline`, `structural`, `marine`, `drilling`, `cathodic-protection`,
`wind`, `reservoir`, `manufacturing`, `geospatial`
*(engineering has the broadest domain surface — subcategories here are expected to grow most)*

**data**: `document-intelligence`, `standards`, `online-resources`, `production`

**platform**: `ai-tools`, `workstations`, `ci-cron`

**business**: `cre-finance`, `gtm`, `website`

**maintenance**: `cleanup`, `refactor`, `testing`

**personal**: `home`, `admin`

> Subcategories are stored as free-text strings. `infer-category.py` prefers existing values
> over minting new ones. New values are auto-documented in `specs/skills/category-taxonomy.md`
> whenever `assign-categories.py --apply` runs (append-only log of when each subcategory first
> appeared and in which WRK item).

## What to Build

### 1. Schema change
- Add `category:` and `subcategory:` fields to WRK frontmatter template (work-queue/SKILL.md Work Item Format)
- Defaults: both `uncategorised`
- Allowed values (7 categories × sub-categories) documented in SKILL.md
- Inference helper returns both fields: `infer-category.py` → `{"category": "engineering", "subcategory": "pipeline"}`

### 2. Category at capture time (all creation paths)

`category:` must be inferred and written **at the moment a WRK item is created**, regardless
of which pipeline created it. Every WRK creation path must be updated:

| Creation path | Where to add inference |
|---------------|----------------------|
| `/work add` (user interaction) | Capture step — infer from user description before writing file |
| `comprehensive-learning` Phase 7 / ActionCandidates | WRK item template in the phase script |
| `workflow-gatepass` Future Work Synthesis (Stage 15) | `future-work.yaml` → WRK creation includes category |
| `close-item.sh` follow-up WRK generation | Add category inference call |
| Any agent that creates WRKs directly | Must call the inference helper before writing |

**Inference helper** — `scripts/work-queue/infer-category.py <title> [body]`:
- Returns one of the allowed category values
- Uses keyword matching (same logic as assign-categories.py)
- Can be called inline from bash or imported as a Python function
- Fallback: `uncategorised` (never errors)

### 3. Bulk-assign script (for existing items)
- `scripts/work-queue/assign-categories.py` — reads each WRK file, infers category from
  keywords in title/body (engineering keywords → `engineering`, etc.), writes `category:` field
- Dry-run by default; `--apply` to write
- Produces a preview table before applying

### 4. INDEX.md grouped view
- `generate-index.py` updated to produce a **Category View** section above the Master Table
- Each category = collapsible section heading with count + priority-sorted item list
- Format per item: `WRK-NNN [PRIORITY] title — status`
- Within each category: HIGH first, then MEDIUM, then LOW; within same priority, sort by WRK ID ascending

### 5. `/work list --by-category` flag
- New flag that renders the category-grouped view instead of flat list
- Default (`/work list`) remains unchanged for backward compatibility
- `--category <name>` narrows to a single category for focused session view

### 6. Session-start surface
- `session-start` skill updated to show top 1-2 unblocked items per category (not just top 3 overall)
- Gives an at-a-glance view of the highest-leverage item in each area
