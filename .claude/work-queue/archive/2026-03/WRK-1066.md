## Mission
Capture the exact version of every AI and build tool on all 3 workstations in a
single machine-readable file, flag divergences, and fix critical mismatches so
"works on dev-primary, fails on dev-secondary" stops being a debugging surprise.

## What
- Script: `scripts/maintenance/ai-tools-status.sh` — reads `scripts/readiness/harness-config.yaml`
  for machine connection details; collects PATH tool versions via explicit-PATH SSH;
  writes `config/ai_agents/ai-tools-status.yaml` (enhanced schema)
- Output: `config/ai_agents/ai-tools-status.yaml` — per-machine reachability + tool status +
  cross-machine drift block + collection_summary
- Drift semantics: missing → BLOCK; major diff → BLOCK; minor diff → WARN; patch → INFO
- Remediation: fix claude/codex/gemini on dev-primary (npm-channel, version-targeted from dev-secondary)
- Schedule: weekly cron (Sunday 03:15) in both `crontab-template.sh` and `setup-cron.sh`

## Why
WRK-1050/1051 already surfaced Codex config divergence. Tool version drift (claude CLI,
codex, gemini) causes silent failures that appear as flaky tests or broken scripts —
time-consuming to diagnose without a machine-readable parity record.

## Acceptance Criteria
- [ ] `harness-config.yaml` has `linux_reachable: false` on licensed-win-1
- [ ] `ai-tools-status.sh` reads `harness-config.yaml`; uses PATH-explicit SSH + `npm list -g`
- [ ] `ai-tools-status.yaml` has `collection_summary`, per-tool `{raw,semver,status}`, `drift` block
- [ ] Drift computed only across `reachable: true` machines
- [ ] 0 BLOCK drifts for claude/codex/gemini post-remediation; audit-only tools recorded as exceptions
- [ ] `setup-cron.sh` ENTRIES + `crontab-template.sh` comment added (Sunday 03:15)
- [ ] licensed-win-1 `reachable: false`; WRK held open until Phase 2 (local Windows session)

## Plan
Route A (simple). Revised per 10 rounds of Codex cross-review.

1. **`scripts/readiness/harness-config.yaml`** — add `linux_reachable: false` to licensed-win-1 block.

2. **`scripts/maintenance/ai-tools-status.sh`** — pure collector/analyzer:
   - Reads `harness-config.yaml` for connection details (no hardcoded hosts)
   - For each machine: if `ssh_target` non-null and `linux_reachable` not false → SSH
   - SSH uses explicit PATH: `ssh <target> "PATH=\$HOME/.npm-global/bin:\$HOME/.local/bin:\$PATH <cmd>"`
   - Version extraction: `npm list -g --depth=0 --json` (same explicit PATH) parsed with
     `uv run --no-project python -c "..."` (repo rule: no bare python3)
   - `status` (per-machine): `ok | missing | error | unparsable`
   - `severity` (cross-machine): `info | warn | block` — computed over `reachable: true` only
   - Exit 0 always except YAML write failure or local command crash → exit 1
   - Overwrites `config/ai_agents/ai-tools-status.yaml`

3. **`config/ai_agents/ai-tools-status.yaml` schema** — `last_updated`, `collection_summary`
   `{total, reachable, unreachable, unreachable_machines}`, per-machine `{reachable, tools}`,
   per-tool `{raw, semver, status}`, top-level `drift` per tool `{severity, machines, note}`.

4. **Remediation on dev-primary** (npm-channel: claude, codex, gemini only):
   - Query dev-secondary installed versions via `npm list -g --depth=0 --json` (explicit PATH)
   - Install matching: `npm install -g "<pkg>@<version>"` for each
   - Re-run audit; expect 0 BLOCK drifts for these three tools
   - `uv`, `gh`, `python3`, `node`, `git`: audit-only; drifts recorded as WARN/exceptions
   - licensed-win-1: deferred; WRK stays open until local Windows session

5. **Cron: both code files updated** (Sunday 03:15, `full` role):
   - `crontab-template.sh`: add comment entry
   - `setup-cron.sh`: add to ENTRIES array with expanded `${HUB_Q}` and `${LOG}` paths
