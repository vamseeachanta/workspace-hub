## Mission

Prevent /whats-next from showing actively-executing WRK items as "high-ready" when
claim-item.sh was not run, eliminating the cross-wired confusion this causes.

## What

When a session begins executing a pending item without running `claim-item.sh`, the
file stays in `pending/` with `status: pending`. The item then appears in ★ HIGH
PRIORITY in `/whats-next` instead of ▶ WORKING — misleading other sessions and the
user.

Holistic fix across three layers:

1. **Detection** — `whats-next.sh` checks each pending item for a `session-lock.yaml`
   in `assets/WRK-NNN/evidence/` and surfaces those as a distinct `⚠ IN-PROGRESS
   UNCLAIMED` section between WORKING and HIGH PRIORITY.

2. **Auto-claim guard** — `start_stage.py` (Stage entry) verifies the item is in
   `working/` before proceeding; if still in `pending/`, it either auto-runs
   `claim-item.sh` or exits with a clear error: "Claim this item first:
   `bash scripts/work-queue/claim-item.sh WRK-NNN`".

3. **Workflow enforcement** — annotate the work-queue-workflow SKILL.md Stage 8
   contract to make claim-before-execute a hard gate, not a suggestion.

## Why

- Users see WRK-1091 in "high ready" while it's actively executing — causes confusion
  about what's available vs. what's in flight.
- Cross-session collision risk: another session may pick up the same item.
- The fix is low-risk (read-only detection in whats-next.sh, soft guard in
  start_stage.py) and high signal.

## Acceptance Criteria

- [ ] `whats-next.sh` shows `⚠ IN-PROGRESS UNCLAIMED` section for pending items with
  an active `session-lock.yaml`
- [ ] `start_stage.py` warns or errors when item is not in `working/` at stage entry
- [ ] `work-queue-workflow/SKILL.md` Stage 8 marks claim as hard gate
- [ ] Existing tests pass; new test covers the detection logic
- [ ] No false positives: stale/old session-lock.yaml files do not trigger the warning
  (check lock `session_pid` or `locked_at` recency)
