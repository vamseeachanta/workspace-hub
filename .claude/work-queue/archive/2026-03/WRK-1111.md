# feat(context): @ File Reference Pipeline — Clean Context for Every Session

## Mission

Use Claude Code's `@file` mention syntax in CLAUDE.md to auto-include high-value reference
files in every session, eliminating the recurring cost of manually pointing Claude at
schemas, dependency maps, and config files that are relevant to nearly every request.

## Why

Course learning: mentioning `@prisma/schema.prisma` in CLAUDE.md causes that file's
contents to be included automatically in every request — Claude can answer schema questions
immediately without a search-and-read cycle. This is a low-effort, high-leverage context
optimisation that compounds across every session.

## What

### Phase 1: Audit — identify high-value @ reference candidates

For each repo, identify files that:
- Are referenced repeatedly across many sessions (schemas, dep maps, config)
- Change infrequently (stable enough to be safe as always-on context)
- Are small enough not to bloat every request (target ≤200 lines each)

Candidates:
| Repo | File | Why always-on? |
|------|------|----------------|
| workspace-hub | `config/deps/cross-repo-graph.yaml` | Dep routing for every cross-repo decision |
| workspace-hub | `config/onboarding/repo-map.yaml` | Repo capability map, referenced in every session-start |
| workspace-hub | `.claude/skills/workspace-hub/workstations/SKILL.md` | Machine routing for every WRK item |
| assethold | `prisma/schema.prisma` | DB schema for every data model question |
| digitalmodel | `src/geometry/circle.py` | Gatepass reference ("circle math") |
| worldenergydata | `src/worldenergydata/modules/__init__.py` | Backward-compat shim — preserve signal |

### Phase 2: Size + stability check

For each candidate:
- Measure line count — reject if >200 lines (bloat risk)
- Check git log frequency — reject if changed >2×/month (staleness risk)
- Flag as `@ mention` (include always) or `note-only` (just document its path in CLAUDE.md)

### Phase 3: Add @ references to CLAUDE.md files

For each approved candidate, add a context block to the relevant repo's CLAUDE.md:

```markdown
## Key References
The dependency graph is defined in @config/deps/cross-repo-graph.yaml —
reference it for any cross-repo routing decision.
```

Keep additions concise (1-2 sentences per file). CLAUDE.md ≤20 line limit applies —
if the repo's CLAUDE.md is already at the limit, add references to the linked skill/doc
file instead (e.g. the repo's AGENTS.md or a dedicated `context.md`).

### Phase 4: Validate

Run a session in each modified repo and confirm:
- Claude answers a relevant question about the @ referenced file without needing to read it
- Context load feels appropriate (no bloat complaints, response quality maintained)

## Acceptance Criteria

- [ ] Candidate audit table completed (all 6 repos assessed)
- [ ] Size + stability check done for each candidate
- [ ] @ references added to CLAUDE.md (or linked doc) for all approved candidates
- [ ] CLAUDE.md ≤20 line limit respected in all repos
- [ ] Validation pass: at least one question answered per repo using auto-included context
- [ ] `docs/context-pipeline.md` written documenting the @ reference decisions and rationale

## TDD Approach

- Script to check CLAUDE.md line counts across all repos (extend existing validate-file-placement checks)
- Manual spot-check: open each modified CLAUDE.md and confirm @ syntax is correct
- Cross-review (Gemini): verify candidate selection rationale is sound

## Non-goals

- Adding @ references to every possible file — only files that are truly always-relevant
- Modifying how Claude Code parses @ mentions — this is a configuration task only
