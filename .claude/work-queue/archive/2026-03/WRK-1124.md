## Mission

`whats-next.sh` must surface all WRK items that are actively in-progress — including those in
pre-claim stages 1–7 — so the full panorama of open work is visible at a glance.

## What

- **`whats-next.sh` — checkpoint detection**: detect items in `pending/` that have a
  `checkpoint.yaml` in `assets/WRK-NNN/` and surface them in the IN-PROGRESS UNCLAIMED
  bucket (alongside the existing `session-lock.yaml` path)
- **`whats-next.sh` — Stage + Status columns**: add `Stage N` and `Status` columns to ALL
  work item table rows; source from `checkpoint.yaml` (`current_stage`) and gate type:
  - `WAITING` — item is at a hard gate (stages 1, 5, 7, 17) awaiting human approval
  - `START` — item is actively running at a non-gate stage
  - `READY` — no checkpoint; item has not been started yet
- **`whats-next.sh` — orchestrator PID**: for WORKING and UNCLAIMED rows, emit `session_pid`
  from `evidence/session-lock.yaml` so the operator can identify the live process
- **Work-queue-workflow skill (session start step)**: write/refresh `session-lock.yaml` when
  resuming a WRK item so the lock-based path also fires going forward

## Why

WRK-1059 (Stage 6) and WRK-1084 (Stage 2) are actively in-progress but appear in the MEDIUM
ready list because `whats-next.sh` only checks for `session-lock.yaml`. Without stage+status
columns and PID visibility, the operator has no panoramic view of all open work and cannot
quickly identify which processes own which items.

## Acceptance Criteria

1. Items with `checkpoint.yaml` in `assets/WRK-NNN/` and `status: pending` appear in the
   IN-PROGRESS UNCLAIMED section of `whats-next.sh` output
2. All table rows show `Stage N` (e.g. `Stage 6`) and `Status` (WAITING/START/READY);
   items with no checkpoint show `—` for stage and `READY` for status
3. WORKING and UNCLAIMED rows show `PID <n>` sourced from `evidence/session-lock.yaml
   → session_pid`; omitted gracefully if lock absent or field missing
4. Items already in `working/` are unaffected by the checkpoint detection change
5. Items in `pending/` with no checkpoint remain in MEDIUM/HIGH as before
6. `session-lock.yaml` write added to the session-resume path in work-queue-workflow skill

## Out of Scope

- Changing the WORKING section (that remains `working/` folder only)
- Modifying `claim-item.sh` or stage gate contracts
