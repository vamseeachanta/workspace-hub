# feat(work-queue): Centralized WRK Status Index

## Mission

Maintain a single machine-readable index (`wrk-status-index.json`) that is written at
source by workflow scripts — so `whats-next.sh` and any other consumer reads ground-truth
status/machine/metadata without scanning individual WRK files.

## Problem

`whats-next.sh` derives status by scanning file locations (`pending/`, `working/`) and
reading individual frontmatter fields. When a parallel session updates a WRK file's
`status:` field but doesn't move the file (or vice versa), the display is wrong. There
is no reliable single source of truth for current WRK state.

## What

### 1. Index file: `.claude/work-queue/wrk-status-index.json`
Machine-readable JSON, keyed by WRK ID:
```json
{
  "WRK-1143": {
    "status": "working",
    "machine": "ace-linux-1",
    "priority": "high",
    "category": "harness",
    "subcategory": "work-queue",
    "title": "feat(work-queue): centralized WRK status index",
    "last_updated": "2026-03-12T07:30:00Z",
    "updated_by": "claim-item.sh"
  }
}
```

### 2. Helper: `scripts/work-queue/update-wrk-index.sh <WRK-ID> <status>`
- Idempotent upsert into `wrk-status-index.json`
- Reads priority/category/title from the WRK file
- Writes atomically (temp file + mv)
- Records `machine` (hostname), `last_updated` (ISO timestamp), `updated_by` (caller script name)

### 3. Wire into source scripts at every status transition
| Script | Transition | Index update |
|--------|-----------|------|
| `claim-item.sh` | pending → working | status=working |
| `close-item.sh` | working → done | status=done |
| `archive-item.sh` | done → archived | status=archived |
| `generate-index.py` | full rebuild | reconcile all entries |

### 4. Bootstrap: `scripts/work-queue/rebuild-wrk-index.sh`
- Scans `pending/`, `working/`, `blocked/`, `archive/` to populate index from current state
- Idempotent; safe to re-run at any time
- Called by `generate-index.py` as reconcile step

### 5. Update `whats-next.sh`
- Read status + machine from index as primary source
- Fall back to file-location heuristic only for items absent from index
- `--debug` flag shows `(index)` vs `(scan)` source per item

## Why

- **Deterministic**: written by scripts at source, not inferred by scanning
- **Cross-session visible**: all sessions on the same machine read the same file
- **Fast**: single JSON parse vs. N `grep` calls across hundreds of files
- **Auditable**: `updated_by` field traces every status change to its source script

## Acceptance Criteria

- [ ] `rebuild-wrk-index.sh` produces valid `wrk-status-index.json` from current queue state
- [ ] `update-wrk-index.sh` upserts correctly; idempotent on repeat calls
- [ ] `claim-item.sh` updates index to `status=working` on successful claim
- [ ] `close-item.sh` updates index to `status=done`
- [ ] `archive-item.sh` updates index to `status=archived`
- [ ] `whats-next.sh` reads status from index; parallel WRKs started scenario shows correctly
- [ ] `generate-index.py` calls rebuild as reconcile step
- [ ] 6+ TDD tests in `tests/work-queue/test_wrk_index.py`

## Out of Scope

- Real-time cross-machine sync (each machine owns its own index)
- Replacing `INDEX.md` (human-readable listing stays separate)
