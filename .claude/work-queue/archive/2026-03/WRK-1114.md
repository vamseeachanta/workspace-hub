## Mission
Partition WRK ID space by machine so concurrent captures on different machines never collide.

## What
Add a machine-specific ID floor convention to `next-id.sh` and document it in
`state.yaml` + the work skill. Each machine owns a non-overlapping numeric range and
re-allocates by bumping its floor when it nears the ceiling.

## Why
dev-primary and licensed-win-1 both pull from the same git repo and run `next-id.sh`
independently. Without coordination, two machines can emit the same WRK-NNN, causing
silent overwrites on the next `git pull`.

## Acceptance Criteria
- [ ] `config/work-queue/machine-ranges.yaml` defines the canonical partition table
- [ ] `next-id.sh` reads the table, detects `$HOSTNAME`, and enforces a minimum floor
      for the current machine (if MAX_ID < floor → start from floor)
- [ ] state.yaml gains a `machine_ranges` comment block pointing to the config file
- [ ] work-queue skill and `next-id.sh` header both document the convention and the
      re-allocation policy ("bump floor when within 50 of ceiling")
- [ ] Tested: simulate dev-primary (floor 1, ceiling 4999) and licensed-win-1 (floor 5000)
      returning non-overlapping IDs
- [ ] No existing WRK IDs are renumbered

## Plan

**Initial partition table** (`config/work-queue/machine-ranges.yaml`):

| Machine | Floor | Ceiling | Notes |
|---------|-------|---------|-------|
| dev-primary | 1 | 4999 | default; current IDs ~1113 |
| licensed-win-1 | 5000 | 9999 | Windows machine |
| dev-secondary | 10000 | 14999 | reserved |
| gali-linux-compute-1 | 15000 | 19999 | reserved |

**Step 1** — Write `config/work-queue/machine-ranges.yaml`

**Step 5** — Update work-queue SKILL.md §Workstation Routing with the range table
