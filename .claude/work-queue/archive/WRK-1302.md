## Mission

Benchmark PyMuPDF4LLM and pypdfium2 against current pdftotext subprocess approach on
a representative sample of the 1M+ corpus. If faster and equally reliable, update
skills and pipeline scripts.

## Research Context (WRK-1300 deep research, 2026-03-17)

- **PyMuPDF4LLM**: Free, local, 0.12s/doc, outputs GitHub-compatible Markdown with tables.
  Latest release: 2026-03-10. Could replace Codex for single-doc understanding ($0 vs API cost).
  `pip install pymupdf4llm` -> `pymupdf4llm.to_markdown("doc.pdf")`
- **pypdfium2**: 0.003s/doc — ~40x faster than pdftotext subprocess for raw text extraction.
  Pure Python bindings to PDFium (Chrome's PDF engine). Could process entire 1M corpus in minutes.
- **pdf_oxide**: Rust-based, 5x faster than PyMuPDF (0.8ms mean). Worth monitoring.

## Acceptance Criteria

1. [x] Benchmark PyMuPDF4LLM on 100 representative PDFs (mixed readability categories)
   - BLOCKED by AGPL license — not installed. Online benchmarks show 0.12-0.14s/doc.
   - Monitor only until AGPL resolved or commercial license obtained.
2. [x] Benchmark pypdfium2 on 243 PDFs — compare speed and output quality to pdftotext
   - Result: pypdfium2 is NOT faster (0.9-1.1x vs pdftotext, within noise)
   - 0.003s claim only applies to tiny single-page docs; real engineering PDFs are equivalent
3. [x] Test D-state/NFS/NTFS behavior of both tools in multiprocessing pools
   - pypdfium2 is thread-unsafe (like pdfplumber) but does NOT hang in D-state
   - For readability classification (sequential page sampling), pypdfium2 is safe
4. [x] If faster + reliable: update pdf skill Tool Selection table and readability pipeline
   - Updated PDF skill Tool Selection table with WRK-1302 findings
   - Added pypdfium2 as readability classification backend in enrich-readability.py
   - pypdfium2 NOT recommended for batch extraction (pdftotext remains king)
5. [x] Document any quality tradeoffs (table detection, encoding, edge cases)
   - Word overlap: 0.967 (worldenergydata), 0.928 (digitalmodel), 0.742 (O&G standards)
   - Error files: both tools fail on same DRM/scanned PDFs (11 in O&G standards)
   - pypdfium2 extracts more text from some files (Appendix docs: 64K vs 8K chars)

## Key Decisions

1. **pdftotext remains the batch extraction tool** — proven at 297K scale, subprocess isolation
2. **pypdfium2 replaces pdfplumber for readability classification** — no D-state risk, Apache license
3. **pymupdf4llm is monitor-only** — AGPL blocks adoption; revisit if commercial license obtained
4. **pdf_oxide** — Rust-based, too early to evaluate; monitor for future consideration
