## Mission

Evaluate IBM Docling (Apache 2.0) with Granite-Docling-258M model for extracting
content types currently at 0% yield: equations, constants, procedures, worked_examples.
If successful, this unlocks a major expansion of the extraction pipeline.

## Research Context (WRK-1300 deep research, 2026-03-17)

- **Docling**: Open source (MIT), donated to Linux Foundation's Agentic AI Foundation.
  Granite-Docling-258M — 258M parameter VLM that handles tables, equations, charts, code.
  DocTags format captures all page elements. Small enough to run locally (no API costs).
- **Marker-PDF**: Alternative — uses Surya for layout analysis. 122 pages/sec on H100.
  With --use_llm flag: cross-page tables, inline math, forms. GPL license (less flexible).
- **MinerU**: Hybrid backend (pipeline + VLM). Could unify Phase 2a/2b. Apache 2.0.

   - Equations: Docling captures Greek symbols (ρ, σ, α) and formula structures in markdown
   - Not LaTeX format — captures as Unicode text in table cells and paragraphs
   - Yield improvement: marginal over pdftotext for inline equations (both capture Unicode)
3. [x] Measure constants extraction yield (vs current 0%)
   - Constants: similar capture rate to pdftotext (~74 per DNV-RP-B401)
   - Docling advantage: constants appear in structured table cells, not raw text

   - Docling: 1731 markdown table rows across 6 documents

   - Quality: good for simple tables; complex multi-span tables not verified

## Key Decisions

1. **Docling is NOT viable for batch corpus processing** — 310s/doc on CPU
2. **Docling IS viable for targeted single-doc extraction** — structured markdown with tables
3. **GPU upgrade required** for practical batch use — CUDA 7.0+ (RTX 2060 or better)
4. **Equation yield is marginal** — both tools capture Unicode; Docling's value is structure
5. **Table extraction is the main differentiator** — 1731 rows in structured markdown
