# WRK-1031 — Single Lifecycle HTML: Embed Full Plan Content Inline, Retire Snapshot Files

## Mission
Rewrite `generate-html-review.py` to support `--stage N --update` mode so the single
`WRK-NNN-lifecycle.html` file captures full plan content (ACs, phases, resource intelligence)
at Stage 4/5, making the separate snapshot files (`plan-draft-review.html`,
`plan-final-review.html`, `implementation-review.html`) redundant and safe to delete.

## What

Currently each WRK produces multiple HTML files that overlap and confuse:
- `WRK-NNN-lifecycle.html` — stage tracker with brief summaries only
- `plan-draft-review.html` — full plan with ACs, phases, resource intelligence
- `plan-final-review.html` — post-cross-review plan snapshot
- `implementation-review.html` — implementation detail snapshot
- `workflow-final-review.html` — close/archive snapshot

This was diagnosed in a user session (2026-03-07): deleting the snapshot files causes data
loss because the lifecycle HTML Stage 4 section only has bullet summaries, not the full
plan content (ACs, detailed phases, gate-pass status). The root cause is that
`generate-html-review.py` lacks a `--stage N --update` mode — agents work around this
by generating separate snapshot files instead of embedding content inline.

## Why

- One URL per WRK — user opens one file and sees everything
- No confusion about which file is canonical
- Gate checkers read one file
- Approval records inline at the stage that required them
- Already the design intent of workflow-html SKILL v1.4.0 (WRK-1028)

## Agentic AI Horizon

- Directly unblocks fully autonomous WRK execution: agents update one file at each stage
  exit rather than maintaining a parallel set of snapshot files with inconsistent coverage
- Once `--stage N --update` exists, `exit-stage.sh` can call it automatically — zero
  human overhead for HTML upkeep

evidence files on disk — the same pattern as the existing `--type` mode, extended to cover
all 20 stages. No HTML parsing or mutation required.

1. Add `generate_lifecycle(wrk_id, output_path)` function alongside `generate_review()`
2. Stage-detection: for each stage 1–20, check canonical evidence file presence to determine
   status (done / active / pending / n/a):

   - S2: `evidence/resource-intelligence.yaml`

   - S5: `evidence/user-review-plan-draft.yaml`

   - S7: `evidence/user-review-plan-final.yaml`

   - S11: `evidence/gate-evidence-summary.json`
   - S12: `variation-test-results.md` or `evidence/test-results.yaml`
   - S13: `evidence/cross-review-impl.md`
   - S14: `evidence/gate-evidence-summary.json` (verify-gate-evidence pass)
   - S15: `evidence/future-work.yaml`
   - S16: `evidence/resource-intelligence-update.yaml`
   - S17: `evidence/user-review-close.yaml`
   - S20: WRK in `archive/`
3. Stage-content renderers (one per stage, reads from YAML/MD source — never from HTML):

   - S5: render from `user-review-plan-draft.yaml` → approval record + Q&A schema block
   - S7: render from `user-review-plan-final.yaml` + cross-review files → verdict table + approval
   - S10/11: render from `evidence/execute.yaml` + `gate-evidence-summary.json` → files table, tests, gates

4. Stage strip chip auto-generated from detection results (done/active/pending/n/a)
5. CLI: `generate-html-review.py WRK-NNN --lifecycle [--output path]` (no --stage flag needed)
6. Remove `--type plan-draft|plan-final|implementation|close` flags entirely (Q3 — deprecated)
7. Unit tests: stage-detection logic; stage-content renderers; full round-trip generation
