# feat(doc-intel): Squeeze Maximum Value from Document Corpus

## Mission

Transform document collections (Drilling Riser + CP as Phase 1) into integrated
`digitalmodel` parametric wrappers, publishing lookup charts open-source and running
detailed analysis on-demand — until each source document is fully absorbed and marked
exhausted.

## Why

The document corpus exists to serve one purpose: eliminate the need to re-read source
documents by integrating their data, analysis, and concepts directly into the codebase.
The GTM target is open-source publication and website deployment. This WRK formalises
the extraction pipeline starting with two domains:

- **Phase 1 domains**: Drilling Riser + Cathodic Protection
- **All remaining domains**: tackled iteratively in follow-on WRKs
- **Lookup charts** → baked into code as tabular constants or interpolation functions;
  sufficient for open-source publish
- **Detailed analysis** → on-demand parametric modules invoked per request
- Once a document's key content is absorbed: mark `exhausted: true` in the ledger;
  source document no longer needed for routine work

### Phase 2 — Document-Verified Tests + Gap-Fill

**Per-document workflow** — execute in order for each selected document:

**Step 1 — Worked example extraction**
Locate the worked example(s) in the document (input values + expected output + source
section). Record in `specs/wrk/WRK-1113/doc-extracts/<doc-ref>.yaml`:
```yaml
doc_ref: 2100-RPT-4005-06
section: "§4.2"
calc: current_demand
inputs: {area_m2: 150.0, current_density_a_m2: 0.025, coating_factor: 0.05}
expected: {value: 0.1875, unit: A, note: "doc states 0.19 A rounded"}
```

**Step 2 — Map to existing route**
Identify which function in `src/digitalmodel/cathodic_protection/` covers this calc.
If found, proceed to Step 3. If missing, go to Step 4.

**Step 3 — Write verification test**
```python
def test_current_demand_from_2100_rpt_4005_section_4_2(self):
    """Verify against worked example: 2100-RPT-4005-06 §4.2."""
    result = current_demand(area=150.0, current_density=0.025, coating_factor=0.05)
    assert result == pytest.approx(0.1875, rel=0.01)
```
If test passes → route is verified. If test fails → diagnose and fix the route.

**Step 4 — Gap-fill (TDD)**
Function missing → write the failing test first (doc example as assertion), then
implement the minimal function following existing module conventions:
typed params, module-level standard constants, `@doc_ref` in docstring.

**Step 5 — Data extraction (legal-gated)**
Extract any tabular data, empirical curves, or reference constants that can be
represented as plain numerical constants (no client identifiers):
- Material properties tables → module-level `dict` or `numpy` array
- Lookup charts → interpolation constants (`scipy.interpolate` or piecewise linear)
- Run `scripts/legal/legal-sanity-scan.sh` on the extract before committing;
  block-severity violations must be resolved (rename generic, remove proprietary refs)

**Step 6 — Parametric sweep test**
For each new or verified function: vary one key input over its physical range and assert
monotonicity or a known trend (e.g. more area → more current demand).

**Step 7 — Analysis improvements**
After verifying the route against the document, note any improvements identified:
- Missing edge-case handling (e.g. coated vs. bare structure branching)
- Better default constants from a newer edition of the standard
- Additional output quantities (e.g. return `CalcResult` with intermediate steps)
Record in `specs/wrk/WRK-1113/improvements-log.md`; implement only if clearly in scope,
otherwise capture as a follow-on WRK.

**Step 8 — Exhaustion marking**
Once all key calcs in the document have a test + any extractable data is absorbed:
set `exhausted: true` + `exhausted_at` + `absorbed_into: [module path]` in
`standards-transfer-ledger.yaml`.

Test file naming: `tests/cathodic_protection/test_<route>_doc_verified.py`
