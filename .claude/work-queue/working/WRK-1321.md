# WRK-1321: Two-Tier Folder-Skill Architecture for Work-Queue

## Problem Statement

Work-queue workflow spread across 3 overlapping skill trees, 20 bare micro-skills, 20 YAML contracts, and 60+ flat scripts. Agent doesn't know which skill to load when. Rules scattered. Scripts have no clear mapping to stages.

Anthropic best practices (Thariq @trq212, 2026-03): "A skill is a folder, not just a markdown file" — folders with scripts, assets, references, on-demand hooks, and gotchas.

## Design

### Tier 1: work-queue-orchestrator/ (ONE folder-skill)
- Lean SKILL.md (~50 lines): stage FSM, R-25/26/27, prep/close contracts
- scripts/: prep_stage.py, close_stage.py, generate-html-review.py, verify-gate-evidence.py
- references/: stage-gate-policy.md, transitions.yaml
- hooks.yaml: always-on "no bypassing stage machinery"
- Orchestrates stage-to-stage. Does NOT execute stage work.

### Tier 2: stage-NN-name/ (20 folder-skills)
- Each stage is a folder: SKILL.md, scripts/, references/, assets/, hooks.yaml, gotchas.md
- Stage-specific scripts co-located (not in global scripts/)
- On-demand hooks active only during that stage
- Contract.yaml from global stages/ into each folder
- Gotchas from 11 documented user complaints + operational lessons

### What Gets Eliminated
- work-queue-workflow/SKILL.md → merged into orchestrator
- workflow-gatepass/ (9 sub-skills) → split into stage folders + orchestrator
- coordination/work-queue/ (15 sub-skills) → reference docs distributed
- Bare stages/stage-NN-*.md → folder-skills
- 60+ flat scripts → distributed to stage folders or orchestrator
