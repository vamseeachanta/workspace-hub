## Execution Notes

- **2026-03-16 ~now**: Running single-threaded retry (not 10 workers). Estimated completion ~9 hours from start (~2026-03-17 early morning).
- Single thread chosen to avoid pdfplumber crash/timeout issues that caused the original 324K errors.

## Mission

Reclassify 324,390 PDFs currently marked readability="error" in the document
index. All files exist on disk — pdfplumber timed out or crashed during the
parallel enrichment run. A retry with robust error handling will reclassify
most as machine/ocr-needed/mixed, unblocking 31% of the corpus.

## Why

The readability enrichment ran 1M docs in 3.1h with 10 parallel workers.
324K PDFs got "error" because pdfplumber exceptions in workers were caught
as generic failures. Zero files are actually missing — they just need retry.

## Plan

### Step 1: Fix enrich-readability.py timeout handling

- Increase per-file timeout from 30s to 60s
- Add per-file error logging (capture exception type)
- Add retry logic: if pdfplumber fails, try single-page sample before giving up
- Filter run: only process records where readability="error"

### Step 2: Run targeted reclassification

```bash
uv run --no-project python scripts/data/document-index/enrich-readability.py \
    --resume --workers 10 --filter-errors-only
```

### Step 3: Analyze remaining errors

After retry, categorize any still-erroring PDFs:
- Encrypted/DRM-protected → readability="drm-protected"
- Corrupted → readability="corrupt"
- Zero-byte → readability="empty"

## Acceptance Criteria

1. [ ] enrich-readability.py has --filter-errors-only flag

3. [ ] Retry run completes on 324K PDFs
4. [ ] Error count reduced by >80% (expect <65K remaining)
5. [ ] Remaining errors categorized (drm/corrupt/empty)
