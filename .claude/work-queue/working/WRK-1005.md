# assess(orchestrators): Cross-Provider Capability Assessment

## What

Consolidate the evidence from WRK-1002 (Claude), WRK-1003 (Codex), and WRK-1004
(Gemini) into a single structured assessment report. Evaluate each orchestrator's
ability to run the full work-queue gatepass lifecycle independently, and identify
strengths, gaps, and recommended routing rules.

This item is purely analytical — no new implementation. Blocked on all three
provider tests completing first.

## Why

Running the three lifecycle tests in WRK-1002/1003/1004 produces comparable evidence,
but the comparison is split across three assets directories. A dedicated assessment
item keeps that synthesis clean, avoids cluttering the individual WRKs, and produces
a single actionable report that feeds into:
- WRK-679 (harness standardization)
- Workstations SKILL.md routing table
- comprehensive-learning Phase 1 orchestrator trend analysis

## Assessment Dimensions

### 1. Lifecycle Completeness
Can each orchestrator run all 8 stages (CAPTURE → ARCHIVE) without manual intervention?

### 2. Gate Compliance
Did each provider pass all `verify-gate-evidence.py` gates on first attempt, or
require remediation?

### 3. TDD Faithfulness
Did each provider write failing tests before implementation (true TDD), or implement
first and backfill tests?

### 4. User-Review Integration
Was the `plan-html-review-final.md` user confirmation correctly triggered and
captured before implementation proceeded?

### 5. Tool / Skill Invocation
Which providers can invoke `/work` skill directly vs. require script fallback?
(See WRK-1000 for Codex limitation)

### 6. Session Log Review
Read and parse the session log files for each provider. Assess:

| Check | Claude | Codex | Gemini |
|-------|--------|-------|--------|
| Log file present | `logs/orchestrator/claude/` | `logs/orchestrator/codex/` | `logs/orchestrator/gemini/` |
| `session_open` event recorded | JSONL `event` field | text scan | text scan |
| `load_assessment` event present | JSONL | n/a | n/a |
| WRK-1002/1003/1004 referenced | by WRK id | by WRK id | by WRK id |
| Tool call count (proxy for effort) | sum from JSONL | line count | line count |
| Gate events logged | plan/tdd/legal/cross-review | grep | grep |
| Session duration | `ts` delta | timestamps | timestamps |
| Errors or retries logged | error fields | grep ERROR | grep ERROR |

Log parsing script: `scripts/work-queue/parse-session-logs.sh WRK-1002 WRK-1003 WRK-1004`
(create if not exists — must handle both JSONL and plain-text log formats)

### 7. Divergence from Spec
Did any provider produce a `calculate_circle` implementation that diverged from
the canonical contract (return keys, precision, edge cases)?
