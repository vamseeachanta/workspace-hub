# feat(session): Extend detect-drift.sh to Parse Codex JSONL Log Format

## Mission

Extend drift detection (WRK-691) to cover Codex sessions by adding a Codex-native
JSONL parser to detect-drift.sh and wiring it into comprehensive-learning Phase 1b.

## What

Codex session logs at `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl` use a different
schema from Claude's logs (`type`/`payload` envelope vs flat `cmd`/`path` fields).
detect-drift.sh currently only handles Claude's format.

### Codex log schema (observed)

```json
{"timestamp": "...", "type": "response_item", "payload": {"type": "message", ...}}
```

Tool calls appear as `type: "function_call"` or inside assistant message content.
Bash commands appear in `payload.function.arguments` or similar nested fields.

### Deliverables

| File | Action |
|------|--------|
| `scripts/session/detect-drift.sh` | ADD `--provider codex` flag; new Python extractor for Codex JSONL schema |
| `scripts/session/tests/test_detect_drift.sh` | ADD tests for Codex log parsing (≥3 cases) |
| `scripts/cron/comprehensive-learning-nightly.sh` | WIRE Phase 1b to also scan yesterday's Codex logs |
| `comprehensive-learning/references/pipeline-detail.md` | UPDATE Phase 1b section with Codex log path |

### Non-deliverable

`.codex/prompts/session-open.md` — not needed; CODEX.md already surfaces all required
gates. Codex has no guaranteed equivalent of Claude's `/session-start` hook.

## Acceptance Criteria

- [ ] `detect-drift.sh --log <codex-rollout.jsonl> --provider codex` runs without error
- [ ] Python extractor identifies python_runtime violations in Codex bash tool calls
- [ ] Tests pass: ≥3 Codex-format test cases in test_detect_drift.sh
- [ ] Phase 1b in nightly script scans Codex logs alongside Claude logs
- [ ] pipeline-detail.md updated to reflect Codex log path and provider flag
